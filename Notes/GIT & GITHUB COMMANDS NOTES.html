Page 2

Configuring Git (one-time setup)

git config --global user.name "Riya Sharma"

What it does: Sets the name attached to every commit you make on this machine, across all repositories.

git config --global user.email "riya@ai.com"

What it does: Sets the email recorded with each commit. GitHub uses this to match commits to your account.

git config --global init.defaultBranch main

What it does: Makes every new repo start with a branch named main instead of the older default master.

git config --global core.editor "code --wait"

What it does: Sets VS Code as the editor Git opens for commit messages, merge messages, and interactive rebase.

git config --list

What it does: Prints every Git setting currently active (global + local), so you can check what is configured.

git config --global alias.co checkout

What it does: Creates a shortcut — typing git co now runs git checkout. Useful for any frequently typed command.

Starting / Downloading a Repository

git init

What it does: Creates a brand-new, empty Git repository in the current folder by adding a hidden .git directory.

git clone <repo-url>

What it does: Downloads a full copy of an existing remote repository — code, branches, and entire commit history — into a new folder.

git clone <repo-url> my-folder

What it does: Same as a normal clone, but saves the project into my-folder instead of the repository's default name.

git clone --depth 1 <repo-url>

What it does: A "shallow clone" — downloads only the latest snapshot, skipping full history. Much faster for huge AI/ML repos.

Checking the Current State

git status

What it does: Shows which files are modified, staged, or untracked right now — the command you run before every commit.

git status -s

What it does: Same information as git status, but in a compact, short-form layout — one line per file.

Staging, Committing & History
SAVING CHANGES · VIEWING DIFFS · READING LOGS
Page 3

Staging Files

git add train.py

What it does: Stages one specific file — moves a snapshot of it into the staging area, ready to be committed.

git add .

What it does: Stages every modified and new file in the current directory and its subfolders, all at once.

git add -p

What it does: Interactively walks through each change in a file and lets you stage only specific chunks, not the whole file.

git restore --staged train.py

What it does: Unstages a file — removes it from the staging area without touching your actual edits.

Committing

git commit -m "feat: add LSTM model"

What it does: Permanently saves all staged changes as a new commit, with the given message describing what changed.

git commit -am "fix: NaN loss bug"

What it does: Stages every already-tracked, modified file and commits in one step — skips a separate git add.

git commit --amend

What it does: Replaces the most recent commit with a new one — lets you fix its message or add forgotten files.

git commit --amend --no-edit

What it does: Adds newly staged files into the last commit while keeping its original message unchanged.

Viewing Differences

git diff

What it does: Shows line-by-line changes in tracked files that are not yet staged, compared to the last commit.

git diff --staged

What it does: Shows exactly what is sitting in the staging area — i.e. what the next commit will contain.

git diff main..feature/lstm

What it does: Compares two branches directly and shows every line that differs between them.

git show a3f2c1d

What it does: Displays the full details and exact changes introduced by one specific commit hash.

Reading History

git log

What it does: Lists all commits on the current branch — hash, author, date, and message — newest first.

git log --oneline --graph --all

What it does: Shows a compact, one-line-per-commit history with a visual branch/merge graph across all branches.

git log --author="Riya"

What it does: Filters the commit history to show only commits made by a specific person.

git log --since="2 weeks ago"

What it does: Filters commit history to a date range — handy for writing a weekly progress report.

git blame model.py

What it does: Shows, line by line, which commit and author last changed each line of a file.

Branching, Merging & Rebasing
CREATING BRANCHES · SWITCHING · COMBINING WORK
Page 4

Creating & Switching Branches

git branch

What it does: Lists all local branches and highlights the one you are currently on.

git branch feature-attention

What it does: Creates a new branch named feature-attention but does not switch to it yet.

git checkout feature-attention

What it does: Switches your working directory to an existing branch, updating all files to match it.

git checkout -b feature-attention

What it does: Creates a new branch and switches to it immediately — one command instead of two.

git switch main

What it does: The newer, dedicated command for switching branches — does the same job as git checkout <branch>.

git switch -c feature-attention

What it does: The modern equivalent of git checkout -b — creates and switches to a new branch.

git branch -d feature-attention

What it does: Deletes a branch, but only if it has already been fully merged — a safety check against losing work.

git branch -D feature-attention

What it does: Force-deletes a branch even if it has unmerged commits. Those commits become hard to recover.

Merging

git merge feature-attention

What it does: Combines the commits from feature-attention into your current branch, integrating both histories.

git merge --no-ff feature-attention

What it does: Forces a merge commit to be created even if a simpler fast-forward merge was possible — keeps branch history visible.

git merge --abort

What it does: Cancels a merge that is stuck in conflicts and returns the branch to its state before the merge started.

git cherry-pick c7d4e8a

What it does: Copies one specific commit from another branch onto your current branch, without merging everything else.

Rebasing

git rebase main

What it does: Moves your current branch's commits so they sit on top of the latest main, producing a straight, linear history.

git rebase -i HEAD~3

What it does: Opens an interactive editor to reorder, edit, squash, or delete your last 3 commits before they're finalised.

git rebase --abort

What it does: Stops an in-progress rebase and restores the branch to exactly how it was before the rebase began.

⚠️

Never rebase a shared branch: git rebase rewrites commit history. If teammates have already pulled the branch, rebasing it creates conflicting histories for everyone. Only rebase branches that exist solely on your machine.

Remote Operations & Stash
SYNCING WITH GITHUB · TEMPORARY SAVES
Page 5

Managing Remotes

git remote -v

What it does: Lists all remote repositories linked to your local repo, along with their URLs (fetch and push).

git remote add origin <url>

What it does: Links a remote repository to your local one, naming it origin so future commands can refer to it by name.

git remote remove origin

What it does: Removes the link to a remote repository. Your local commit history is completely unaffected.

Sending & Receiving Changes

git push origin main

What it does: Uploads your local commits on main to the origin remote, making them visible to your team.

git push -u origin main

What it does: Pushes and also sets origin main as the default upstream, so future plain git push commands know where to go.

git push origin --tags

What it does: Uploads all your local tags (e.g. release versions) to the remote — tags are not pushed automatically.

git pull origin main

What it does: Downloads new commits from the remote and immediately merges them into your current branch — fetch + merge in one step.

git pull --rebase

What it does: Downloads remote commits and rebases your local commits on top of them, instead of creating a merge commit.

git fetch origin

What it does: Downloads new commits from the remote but does not merge them — lets you inspect changes before integrating.

⚠️

git push --force: Overwrites the remote branch's history with your local one, permanently deleting any commits that existed there but not on your machine. Avoid on shared branches like main.

Stashing Uncommitted Work

git stash

What it does: Temporarily shelves all uncommitted changes and gives you a clean working directory, without committing anything.

git stash list

What it does: Shows every stash you currently have saved, each labelled like stash@{0}, stash@{1}.

git stash pop

What it does: Re-applies the most recent stash to your working directory and removes it from the stash list.

git stash apply stash@{1}

What it does: Re-applies a specific older stash, but keeps it in the stash list (does not delete it).

Undoing Changes & Tags

Page 6

Undoing & Resetting

git reset --soft HEAD~1

What it does: Undoes the last commit but keeps all its changes staged, ready to be re-committed differently.

git reset HEAD~1

What it does: Undoes the last commit and unstages its changes, but keeps them in your working directory as edits.

git reset --hard HEAD~1

What it does: Permanently deletes the last commit and all its changes from your files. There is no staging-area safety net here.

git revert a3f2c1d

What it does: Creates a brand-new commit that undoes the changes from a specific past commit — safe for shared/public history.

git restore train.py

What it does: Discards uncommitted edits in a file, resetting it back to its last committed version.

git clean -fd

What it does: Permanently deletes untracked files and folders (e.g. stray temp/log files) that Git has never seen committed.

⚠️

git reset --hard and git clean -fd: Both permanently delete work with no undo. Always run git status first to confirm exactly what will be removed.

Tagging Releases

git tag v1.0.0

What it does: Creates a lightweight tag — a simple, fixed label pointing at the current commit. Good for quick personal markers.

git tag -a v1.0.0 -m "Release v1.0.0"

What it does: Creates an annotated tag — stores the tagger's name, date, and message. The standard choice for official releases.

git tag -d v1.0.0

What it does: Deletes a tag from your local repository (use git push origin :refs/tags/v1.0.0 to also remove it from the remote).

git push origin --tags

What it does: Uploads all local tags to the remote so they're visible to the whole team on GitHub.

GitHub CLI (gh) — Pull Requests & Issues

gh repo create my-ai-app --public

What it does: Creates a new repository directly on GitHub from the terminal — no need to open the browser.

gh pr create --title "Add attention layer"

What it does: Opens a Pull Request from your current branch into main, straight from the command line.

gh pr list

What it does: Lists all open Pull Requests on the repository, with their status and reviewers.

gh pr checkout 42

What it does: Fetches and switches your local repo to the exact branch used in Pull Request #42, for local testing.

gh pr merge 42 --squash

What it does: Merges Pull Request #42 into the base branch, squashing all its commits into one clean commit.

gh issue create --title "NaN loss bug"

What it does: Creates a new GitHub Issue to track a bug, task, or feature request, without opening a browser.

.gitignore — Excluding Files for AI Projects

# Ignore Python build artifacts __pycache__/ *.pyc # Ignore virtual environments venv/ # Ignore large model & data files (track these with DVC instead) *.pt *.h5 *.pkl data/raw/ # Never commit secrets .env *.pem

What this file does: Each line is a pattern Git will never track or stage, no matter how many times you run git add .. Essential in AI projects to keep model weights, datasets, and API keys out of the repository.

GitHub Actions — Minimal CI Workflow

name: Run Tests on: [push] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: pip install -r requirements.txt - run: pytest

What this file does: Saved as .github/workflows/test.yml, it tells GitHub to automatically install dependencies and run your test suite every time code is pushed — catching breakages before they reach main.

DVC — Versioning Large Datasets & Models

dvc init

What it does: Sets up DVC tracking inside a Git repository, alongside your normal Git setup.

dvc add data.csv

What it does: Tracks a large file with DVC instead of Git — Git stores only a small pointer file (data.csv.dvc).

dvc push

What it does: Uploads the actual large data/model files to remote storage (e.g. S3, Google Drive) — not GitHub itself.

dvc pull

What it does: Downloads the large files referenced by DVC pointer files — used right after git clone or git pull.
