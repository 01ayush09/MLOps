Kubernetes Level 101 — What Is Kubernetes?
01
Concept #1

01.What Is Kubernetes & Why Does It Exist?

📖 Definition Kubernetes (often written as K8s — the 8 represents the 8 letters between K and s) is an open-source container orchestration platform that automates the deployment, scaling, management, and healing of containerised applications. It was originally developed by Google and released in 2014, inspired by Google's internal cluster manager called Borg.

The Problem Kubernetes Solves

Imagine you've built a RAG application. It runs perfectly on your laptop in a Docker container. Now imagine 50,000 users are hitting it simultaneously. You need:

  • Many copies of the application running at once (multiple instances or replicas)
  • A way to automatically start a new instance when one crashes
  • A way to route traffic across all instances evenly
  • A way to update the application without downtime
  • A way to scale up (more instances) when traffic is high, and scale down at night
  • A way to allocate GPU resources to the model-serving component specifically

Managing all of this manually across dozens of servers is impossible. Kubernetes does all of it automatically.

Think of Kubernetes as the air traffic control system of your application. Just as air traffic control knows exactly which runway each plane uses, which planes are airborne, when to redirect traffic during delays, and how to handle emergencies — Kubernetes knows exactly which server runs which application, when to restart a crashed container, how to route incoming requests, and how to handle failures without human intervention.

The Three Core Promises of Kubernetes

Self-Healing If a container crashes, K8s automatically restarts it. If a server dies, K8s moves all its containers to other servers. Applications stay up without human intervention.
Scaling K8s can automatically add more copies of your app when traffic increases and remove them when traffic drops — saving money and maintaining performance.
Declarative Management You tell K8s what you want (e.g., "run 5 copies of my RAG app"), not how to do it. K8s figures out how and keeps the system in that desired state forever.

Why AI Engineers Specifically Need Kubernetes

Kubernetes Level 104 — Pods

Concept #4

04.Pods — The Atomic Unit of Kubernetes

📖 Definition A Pod is the smallest deployable unit in Kubernetes. It represents a single instance of a running process in your cluster. A Pod encapsulates one or more containers (usually one), their shared storage, shared network (they share an IP address), and specifications for how to run them.

Why Pods and Not Just Containers?

You might wonder: if Kubernetes manages containers, why introduce a "Pod" concept? The reason is that some applications naturally need multiple tightly-coupled containers to run together. For example, a web server container might need a logging sidecar container that reads its logs and ships them elsewhere. These two containers share the same network and storage — they are a single deployable unit, which is exactly what a Pod represents.

However, for most AI engineering work, each Pod contains exactly one container. Think of a Pod as a "wrapper" around your Docker container that gives it a Kubernetes identity, IP address, and lifecycle.

Kubernetes Level 105 — Nodes & Clusters
05
Concept #5

05.Nodes & Clusters

📖 Definitions A Node is a physical or virtual machine in your Kubernetes cluster that runs Pods. A Cluster is the complete set of all nodes (Control Plane + Worker Nodes) managed by a single Kubernetes installation. Every cluster has at least one node.

The Node as a Worker Machine

Think of a Node as a physical office worker's desk — it has a fixed amount of resources (CPU, memory, disk, possibly GPU). The Kubernetes Scheduler is the manager who decides which desk each piece of work (Pod) gets assigned to, based on available space.

Each Worker Node runs three key components: kubelet (the agent that communicates with the Control Plane), container runtime (usually containerd or Docker, which actually runs the containers), and kube-proxy (manages network rules for Pod communication).

How the Scheduler Chooses a Node

When a new Pod needs to be placed, the Scheduler goes through a two-phase process:

Scheduler Decision Process — Filtering then Scoring
Phase 1: Filter Eliminate nodes that cannot run the Pod: not enough CPU, not enough memory, node has a taint the Pod can't tolerate, node doesn't have required hardware (e.g., GPU). This narrows down from all nodes to a feasible set.
Phase 2: Score Among feasible nodes, score each one using priority functions: prefer nodes with more free resources, prefer the node that already has the container image cached, spread Pods across different nodes for fault tolerance. Highest score wins.
Result: Bind Scheduler binds the Pod to the winning node. kubelet on that node receives the Pod spec and starts the container. The binding is stored in etcd.

Kubernetes Level 106 — Deployments
06
Concept #6

06.Deployments — Managing Pods at Scale

📖 Definition A Deployment is a Kubernetes object that manages a group of identical Pods. You tell the Deployment what you want (e.g., "run 3 replicas of my RAG app container"), and it continuously ensures that exact state exists in the cluster — creating, deleting, or replacing Pods as needed.

Why You Almost Never Create Pods Directly

If you create a Pod directly (without a Deployment), and that Pod crashes or the node it's on fails — the Pod is gone forever. Nobody recreates it. You would have to notice and manually restart it.

A Deployment solves this: it watches its Pods at all times and automatically recreates any that disappear. This is called self-healing and is one of the most important features of Kubernetes.

A Deployment is like a staffing agency contract. You tell the agency "I always need 3 developers working on Project X." If one developer quits, the agency automatically hires a replacement. You don't have to notice or react — the agency (Deployment) ensures the count is always 3.

The Desired State Model — The Heart of Kubernetes

Kubernetes operates on a desired state model. You declare what you want (3 replicas, image v2, 500m CPU) and Kubernetes continuously compares the actual state against the desired state. Any difference triggers automatic corrective action by the Controller Manager.

Desired State Reconciliation Loop — Runs Continuously
Desired State: 3 replicas of my-rag-app:v2
↓
Controller Manager: Observe Actual State
↓
Actual = 3 replicas ✅
No action needed
Actual = 2 replicas ❌
Create 1 new Pod
↓
Loop repeats every few seconds forever

What Happens During a Deployment Update

When you update a Deployment (e.g., change the Docker image from v1 to v2), the Deployment creates a new ReplicaSet for the new version and gradually scales it up while scaling down the old ReplicaSet. This is the rolling update strategy, and it is why Kubernetes can update applications with zero downtime.

Rolling Update — Old ReplicaSet Scales Down, New Scales Up
StepOld RS (v1) PodsNew RS (v2) PodsTotal
Before update3 running03
Step 12 running1 running3
Step 21 running2 running3
Step 30 (old RS kept empty)3 running3

At every step, at least 3 Pods are serving traffic. Zero downtime achieved. Old RS kept for rollback.

💡 Why Old ReplicaSet Is Kept After Update After a rolling update completes, Kubernetes keeps the old (now empty) ReplicaSet around. This is intentional — if you need to roll back to the previous version, the Deployment simply scales the old ReplicaSet back up and scales down the new one. The rollback is instant because the old Pod spec is still recorded.

Kubernetes Level 108 — Services
08
Concept #8

08.Services — Networking & Service Discovery

📖 Definition A Service is a stable Kubernetes object that provides a fixed network address and DNS name for a group of Pods. Because Pods are ephemeral (they can be created and destroyed at any time, getting new IP addresses), a Service provides the stable "front door" through which traffic reaches your application.

The Pod IP Problem — Why Services Exist

Every Pod gets its own IP address. But Pods die and are replaced constantly, and each new Pod gets a new IP address. If Service B needs to call Service A, it cannot hard-code Service A's IP — that IP might be gone in 5 minutes. A Service solves this by giving Service A a stable virtual IP (ClusterIP) and a DNS name that never changes, regardless of which Pods are behind it.

Kubernetes Level 109 — ConfigMaps & Secrets
09
Concept #9

09.ConfigMaps & Secrets

📖 Definitions A ConfigMap stores non-sensitive configuration data (environment variables, config files) separately from the container image. A Secret stores sensitive data (API keys, passwords, tokens) in an encoded format, injected securely into Pods at runtime. Both solve the same problem: decouple configuration from code.

Why Not Hard-Code Config in Your Image?

If you bake your OpenAI API key, database password, or environment-specific settings directly into your Docker image, you face several problems: you must rebuild the image every time a config value changes, your API keys are visible to anyone with access to the image, and you need different images for dev/staging/production. ConfigMaps and Secrets solve all three.

ConfigMaps are like the employee handbook — general company policies that everyone can read. Secrets are like individual salary information — only the relevant employee and HR can see it, and it's stored in a locked cabinet (encoded/encrypted), not posted on the wall.

Kubernetes Level 111 — Volumes & Persistent Storage

Concept #11

11.Volumes & Persistent Storage

📖 The Storage Problem By default, a Pod's filesystem is ephemeral — when the Pod dies, all data in its filesystem is lost. For an AI application this is catastrophic: your vector index, model weights, uploaded documents, and database files would all disappear. Kubernetes solves this with Volumes.

The Storage Hierarchy

Volume
Basic storage attached to a Pod
Lives as long as the Pod. Shared between containers in a Pod. Deleted when Pod is deleted.
Persistent Volume (PV)
Cluster-level storage resource
Exists independently of any Pod. An administrator (or StorageClass) provisions it. Backed by real storage: AWS EBS, GCP Persistent Disk, NFS, etc.
Persistent Volume Claim (PVC)
A Pod's request for storage
Your application says "I need 10GB of storage." K8s finds an appropriate PV and binds it. Your Pod uses the PVC — it doesn't care about the underlying storage type.

Kubernetes Level 113 — Scaling & Rolling Updates

Concept #13

13.Scaling & Rolling Updates

📖 Two Types of Scaling Horizontal scaling (more Pods = more instances of the same app) vs Vertical scaling (more CPU/memory per Pod). Kubernetes excels at horizontal scaling — you can go from 3 to 30 replicas with a single command, and the Horizontal Pod Autoscaler (HPA) can do it automatically based on metrics.

Manual Horizontal Scaling

kubectl · Scaling Commands
$ kubectl scale deployment my-rag --replicas=10 # Scale to 10 replicas $ kubectl scale deployment my-rag --replicas=1 # Scale down to 1 replica $ kubectl get pods -w # Watch Pods being created/terminated

Horizontal Pod Autoscaler (HPA)

The HPA automatically scales the number of Pod replicas based on CPU utilisation (or custom metrics like requests-per-second). You define a minimum and maximum replica count, and the HPA adjusts between them to meet your CPU target.

Kubernetes Level 114 — Health Checks & Probes

Concept #14

14.Health Checks — Liveness, Readiness & Startup Probes

📖 Definition Probes are periodic health checks that Kubernetes performs on your containers. They allow K8s to know if a container is alive, ready to accept traffic, or still starting up — and to take automatic corrective action (restart, stop sending traffic) based on the results.

The Three Probe Types — Each Solves a Different Problem

Liveness Probe Question it answers: "Is this container still alive?" If the probe fails, K8s restarts the container. Use for detecting deadlocks — situations where the app is running but completely stuck and will never recover without a restart. Example: your RAG app's Python process is alive but the embedding model thread deadlocked.
Readiness Probe Question it answers: "Is this container ready to receive traffic?" If the probe fails, K8s removes the Pod from the Service's load balancer (stops sending traffic to it) but does NOT restart it. Use for apps that need warm-up time — e.g., loading a 13GB language model into GPU memory takes 60 seconds, during which you don't want traffic.
Startup Probe Question it answers: "Has this container finished starting?" Disables liveness and readiness checks until it succeeds. Use for slow-starting apps — prevents K8s from killing a container that's still loading model weights because the liveness probe fired too early.

💡 AI Engineer Best Practice: Always Define Readiness Probes for Model Serving Loading a large language model (LLaMA 70B, Mistral 7B) into GPU memory can take 30–120 seconds. Without a readiness probe, K8s adds the Pod to the Service immediately after container start — sending real user traffic to a server that's still loading. The requests fail. Always use a readiness probe that only passes after the model is fully loaded and able to respond.
Kubernetes Level 115 — Resource Requests & Limits
15
Concept #15

15.Resource Requests & Limits

📖 Definitions A Resource Request is the amount of CPU/memory a container is guaranteed — the Scheduler uses requests to choose which node to place the Pod on (the node must have at least this much free). A Resource Limit is the maximum a container can ever use — if exceeded, CPU is throttled (slowed) and memory causes the container to be killed (OOMKilled).

CPU and Memory Units

CPU Units CPU is measured in millicores (m). 1000m = 1 CPU core. So 500m = half a CPU core, 2000m = 2 cores. For an AI inference pod that does heavy computation, you might set a limit of 4000m (4 cores) per replica.
Memory Units Memory is measured in bytes with SI suffixes. 512Mi = 512 mebibytes, 4Gi = 4 gibibytes, 16Gi = 16 GB. An embedding model might need 2Gi, while a 7B parameter LLM needs 14–28Gi of RAM.

Why Requests and Limits Are Both Important

GPU Resources in Kubernetes

🔑 How Kubernetes Handles GPUs GPUs are treated as extended resources in Kubernetes. A node with 4 NVIDIA A100 GPUs advertises nvidia.com/gpu: 4. Your Pod requests nvidia.com/gpu: 1 in its resource spec, and the Scheduler ensures it lands on a node with a free GPU. GPU resources cannot be shared between Pods (unlike CPU/memory) — one request of 1 GPU = exclusive access to one physical GPU.

Kubernetes Level 117 — Common Mistakes at Level 1
17
Watch Out

17.Common Mistakes at Level 1

❌ Mistake 1: Storing Data in Pod Filesystem What happens: You store your vector database files inside the container's local filesystem. The Pod gets restarted due to a node failure. All your vectors are gone — you must re-index thousands of documents from scratch. Fix: Always use a PersistentVolumeClaim for any data that took time to create or that must survive Pod restarts.
⚠ Mistake 2: Hardcoding API Keys in Docker Images What happens: Your OpenAI API key is in your application code. You push the Docker image to a registry. Anyone with registry access now has your API key. Fix: Always use Secrets to inject sensitive values at runtime. Your code reads from environment variables, never from hardcoded strings.
⚠ Mistake 3: No Readiness Probe on LLM Service What happens: Your model-serving Pod starts, the container is running, K8s adds it to the Service immediately. But the model (15GB) is still loading. Requests flood in and all fail with errors for 90 seconds. Fix: Add a readiness probe that hits an endpoint that only returns 200 after the model is fully loaded.
⚠ Mistake 4: Creating Pods Directly (Not via Deployments) What happens: You run a Pod directly without a Deployment. The pod crashes. Nobody restarts it. Your service is down until you notice. Fix: Almost always use a Deployment (or StatefulSet for stateful apps). Direct Pod creation is only for debugging.
⚠ Mistake 5: No Resource Requests/Limits What happens: A memory leak in one container gradually consumes all memory on a node. Other Pods on that node are killed. The Scheduler had no warning because there were no declared resource needs. Fix: Always set resource requests (so Scheduler can place Pods correctly) and limits (to protect the node from runaway containers).
⚠ Mistake 6: Working in the 'default' Namespace in Production What happens: Your team's dev work and production services all run in 'default'. Someone accidentally deletes a deployment they thought was the dev one — it was production. Fix: Create separate namespaces for dev, staging, and production from day one.
💡 Mistake 7: Ignoring kubectl describe and Events When something isn't working, most beginners only look at kubectl get pods and see "Pending" or "CrashLoopBackOff" without understanding why. The answer is almost always in kubectl describe pod <name> — the Events section at the bottom tells you exactly what went wrong: out of memory, image not found, port conflict, PVC not bound, etc.

Core Concepts

🚢
Kubernetes — Container orchestration. Automates deployment, scaling, healing. The "air traffic control" of your containers.
📦
Container — Isolated, portable app package. Lighter than VMs, starts in seconds, shares host OS kernel.
🧠
Control Plane — API Server + etcd + Scheduler + Controller Manager. The brain that makes all decisions.
💪
Worker Node — Physical/virtual machine running your Pods. Has kubelet agent + container runtime.
⚛️
Pod — Smallest K8s unit. Wraps one (usually) container. Ephemeral — always use Deployments.
🔁
Deployment — Manages Pod count + updates. Desired state. Self-healing. Rolling updates. Rollback.
🔢
ReplicaSet — Underneath Deployment. Ensures exact N replicas exist. Created automatically.
🌐
Service — Stable IP + DNS for a group of Pods. Types: ClusterIP, NodePort, LoadBalancer.
⚙️
ConfigMap — Non-sensitive config injected into Pods as env vars or files. No rebuild needed.
🔐
Secret — Sensitive data (API keys, passwords). Base64-encoded. RBAC-controlled access.
🏠
Namespace — Logical partition of a cluster. Use per-environment (dev/staging/prod) isolation.
💾
PVC — Persistent storage that survives Pod restarts. Required for databases, vector indexes.
❤️
Probes — Liveness (restart if stuck), Readiness (traffic if ready), Startup (wait for slow start).
📊
Resources — Request = scheduled guarantee. Limit = hard cap. CPU in millicores, memory in Mi/Gi.
