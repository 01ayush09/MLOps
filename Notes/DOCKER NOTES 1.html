Docker for AI Engineers

Images · Containers · Dockerfile · Port Mapping · Volumes

Docker packages an application together with its exact runtime, libraries, and system dependencies into a single portable unit, so it behaves identically on a laptop, a training server, and a production cluster. For AI engineers this solves a very specific pain: a model that trains fine locally but breaks in the cloud because of a mismatched CUDA version, a missing system library, or a different Python build. Docker removes that gap by shipping the entire environment, not just the code.

  • Reproducibility — the same image runs identically on every machine; no more "works on my machine".
  • Isolation — each project gets its own dependency tree, so a CV pipeline and an LLM service never collide.
  • Portability — move a training job or inference API from a laptop to GPU servers or the cloud unchanged.
  • Speed — containers start in seconds because they share the host's kernel instead of booting a full OS.

Docker Field Notes01 · Docker Image
Concept 01

Docker Image

The read-only blueprint your containers are built from

Definition. A Docker image is a read-only, immutable template that bundles an application's code, runtime, system libraries, environment variables, and configuration into one package. It is the blueprint — nothing actually runs until the image is started as a container (page 03).

Key Characteristics

  • Layered — an image is a stack of read-only layers; each layer captures the filesystem changes from one build instruction (page 04 explains this in detail).
  • Immutable — once built, an image's layers never change. To change behaviour you build a new image.
  • Cached & shared — identical layers are stored once and reused across many images, saving disk space and speeding up builds.
  • Addressed by name and ID — referenced as repository:tag (e.g. pytorch/pytorch:2.3-cuda12.1) and uniquely by a SHA-256 content digest.
  • Portable — pushed to and pulled from a registry (Docker Hub, AWS ECR, GCP Artifact Registry, or a private registry) so any machine can fetch the exact same bytes.

Image vs. Registry

  • A local image cache lives on your machine after you build or pull an image.
  • docker push uploads an image to a registry; docker pull downloads one.
  • A registry is just versioned storage for images — think of it as "GitHub for images".
  • Base images (e.g. python:3.11-slim, nvidia/cuda) are themselves images you pull and build on top of.

How an Image Is Composed

Fig. 2 — A Docker image is simply its layers stacked and tagged together.
Why it matters for AI engineers Pin the exact Python, CUDA, and library versions your model needs into an image once — then that exact environment is reproducible on every teammate's laptop, every training node, and the production server. No more "it trained fine for me but not in the cluster".

Docker Field Notes02 · Container
Concept 02

Container

A running, isolated instance of an image

Definition. A container is a running (or stopped) instance of an image. Docker takes the image's read-only layers and adds one thin writable layer on top. That writable layer is where the process can write files, logs, or temp data while it runs — and it disappears when the container is removed.

How Isolation Works

  • Namespaces — the Linux kernel feature giving each container its own view of processes (PID), network (NET), filesystem mounts (MNT), hostname (UTS), and inter-process communication (IPC).
  • Cgroups — control groups limit how much CPU, memory, and I/O a container can consume, so one container can't starve the host or its neighbours.
  • Shared kernel — unlike a virtual machine, a container does not boot its own OS kernel; it shares the host's kernel, which is why containers start in seconds and use far less memory than VMs.

Lifecycle

One Image, Many Containers

Fig. 3 — Multiple isolated containers can run from the same image simultaneously, each with its own writable layer.
Why it matters for AI engineers Run several experiments (different hyperparameters, different library versions) at once from the same base image, each in its own container, without them ever touching each other's files or processes.

Docker Field Notes03 · Dockerfile
Concept 03

Dockerfile

The text recipe Docker reads to build an image

Definition. A Dockerfile is a plain-text file containing an ordered list of instructions. Running docker build reads it top to bottom and executes each instruction inside a temporary build container, committing the filesystem changes as a new layer after every step — the result is the final image (page 02).

Annotated Example

Dockerfile
1FROM python:3.11-slim 2WORKDIR /app 3COPY requirements.txt . 4RUN pip install -r requirements.txt 5COPY . . 6EXPOSE 8000 7CMD ["python", "app.py"]
  1. FROM — chooses the base image every later instruction builds on (here, a slim Python runtime).
  2. WORKDIR — sets the working directory for every instruction that follows; creates it if missing.
  3. COPY requirements.txt . — copies only the dependency list first, on purpose (see caching tip below).
  4. RUN — executes a command during the build and bakes its result into a new layer.
  5. COPY . . — copies the rest of the application source into the image, after dependencies are installed.
  6. EXPOSE — documents which port the app listens on; it does not publish the port (page 05).
  7. CMD — the default command run when a container starts from this image.

Build Context & .dockerignore

  • The folder you run docker build from is the build context — Docker sends its files to the daemon so COPY/ADD can use them.
  • A .dockerignore file excludes things like .git, __pycache__, datasets, or model checkpoints from being sent — keeping builds fast and images small.

Layer Caching Tip

  • Docker reuses a cached layer if the instruction and its inputs are unchanged.
  • Order instructions from least to most frequently changing: copy requirements.txt and install dependencies before copying application code.
  • Result: editing your Python code no longer triggers a slow re-install of every ML library.
Why it matters for AI engineers A Dockerfile makes your training/serving environment version-controllable — review it in a pull request just like code, so "what dependencies does this model need" is never tribal knowledge.

Concept 04

Port Mapping

Letting traffic from outside reach a process inside a container

Definition. By default a container runs in its own isolated network namespace — nothing outside it can reach a port the app listens on. Port mapping (a.k.a. "publishing a port") creates a forwarding rule from a port on the host machine to a port inside the container, using the -p host_port:container_port flag on docker run.
Fig. 4 — -p 8080:5000 forwards requests hitting the host's port 8080 to port 5000 inside the container.

Rules to Remember

  • EXPOSE in a Dockerfile is only documentation — it does not publish a port by itself.
  • -p host:container is what actually opens the door from outside.
  • -P (capital) publishes every EXPOSEd port to a random free host port.
  • You can map several ports in one command, repeating -p for each pair.

Good Practice

  • Bind to localhost only when you don't want the service reachable from other machines: -p 127.0.0.1:8080:5000.
  • Host and container ports don't need to match — only the mapping matters.
  • Use docker port <container> to check what's actually published.
Why it matters for AI engineers When you containerize a FastAPI/Flask model-serving endpoint, port mapping is what lets a teammate, a load balancer, or your own browser actually call /predict on it.

Docker Field Notes05 · Volumes
Concept 05

Volumes

Persistent storage that survives a container's life

Definition. A container's writable layer is ephemeral — it is deleted the moment the container is removed. A volume is storage managed by Docker that lives outside any single container's lifecycle, so data (datasets, model checkpoints, database files) survives container restarts, removals, and upgrades, and can even be shared between containers.
Fig. 5 — Bind mounts point at a host path you choose; named volumes are storage Docker manages for you. Both outlive the container.

Types of Mounts

  • Named volume — created and tracked by Docker (docker volume create), stored under Docker's own directory; best for app/database data you don't need to browse directly.
  • Bind mount — maps a specific folder/file from the host straight into the container; best for mounting source code or datasets you're actively editing on the host.
  • tmpfs mount — stored only in host memory, never written to disk; useful for sensitive, very short-lived data.

Good Practice

  • Never bake large datasets into the image itself — mount them as a volume instead, so the image stays small and fast to build/pull.
  • Mount a volume for model checkpoints so a crashed/restarted training container resumes without losing progress.
  • Multiple containers can mount the same named volume to share data safely.
Why it matters for AI engineers Mount your dataset directory and a checkpoints folder as volumes — your training container can be deleted and recreated freely while the dataset and progress stay safely on disk.
