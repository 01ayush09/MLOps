Docker Networks

CONCEPT 01
1
What is a Docker Network?

Docker networking allows containers to communicate with each other, with the host machine, and with the outside world. Think of it like a virtual LAN (Local Area Network) that Docker manages automatically. When you run a container, it needs to talk to other services — a web app talks to a database, an AI model server exposes an API endpoint, and so on. Docker networks make this possible in a controlled, secure way.

💡 Why it matters for AI Engineers When you run inference servers (e.g. Ollama, TGI, vLLM) alongside API gateways and monitoring stacks, Docker networks let these containers talk privately without exposing internal ports to the internet.

Network Driver Types

🔗 Bridge
Default. Isolated virtual network on the host. Containers on the same bridge can communicate. Best for single-host setups.
🖥️ Host
Container shares the host's network stack directly. No isolation. Lowest latency — useful for high-performance inference.
🚫 None
No networking at all. Container is fully isolated. Used for batch jobs that need no network access.
🌍 Overlay
Spans multiple Docker hosts (Docker Swarm). Enables distributed, multi-node container communication.
🪪 Macvlan
Assigns a real MAC address to a container. Makes it appear as a physical device on the network.
🔌 IPvlan
Like Macvlan but shares the parent MAC. Better for environments where the switch limits MACs per port.

DNS Resolution in Docker

Docker runs an embedded DNS server at 127.0.0.11. On user-defined networks, containers can reference each other by container name — no need to track IP addresses. For example, a Flask app can connect to postgres:5432 simply because both containers share the same custom network.

✅ User-Defined Networks Always create custom networks instead of using the default bridge. You get automatic DNS, better isolation, and containers can be attached/detached while running.
⚠️ Port Publishing -p host:container exposes a container port to the host. Without this, services are only reachable inside the Docker network, not from your browser or external clients.

Docker Compose

CONCEPT 02
2
What is Docker Compose?

Docker Compose is a tool for defining and running multi-container applications using a single YAML file (docker-compose.yml). Instead of typing long docker run commands with dozens of flags, you describe your entire stack — web server, database, cache, model server — declaratively in one file, then bring it all up with a single command.

💡 AI Engineer Use Case A typical ML deployment stack: FastAPI inference server + PostgreSQL metadata store + Redis feature cache + Prometheus metrics. With Compose, the entire stack launches in one command: docker compose up.

Key Compose Concepts

ServicesEach container in your stack is a "service." Each service has its own image, ports, environment, volumes, and dependencies.
NetworksCompose auto-creates a default network so all services can communicate by name. You can also define custom networks.
VolumesPersistent data storage. Databases store data in volumes so it survives container restarts and recreations.

Anatomy of docker-compose.yml

version: "3.9"
services:
  api:                         # service name = DNS hostname
    image: myapp:latest
    build: .                   # build from local Dockerfile
    ports:
      - "8000:8000"            # host:container port mapping
    environment:               # env vars injected into container
      - DATABASE_URL=postgresql://db:5432/mydb
    depends_on:                # start order dependency
      - db
    networks:
      - appnet
  db:
    image: postgres:15
    volumes:
      - pgdata:/var/lib/postgresql/data  # named volume
    networks:
      - appnet
volumes:
  pgdata:                      # declare named volume
networks:
  appnet:                      # declare custom network
    driver: bridge
🔄 depends_on vs healthcheck depends_on only waits for the container to start, not for the service inside to be ready. Use healthcheck in Compose v3+ for proper readiness gating.
📁 .env Files Compose automatically reads a .env file in the same directory. Use this to manage secrets like API keys, database passwords — never hard-code them in the YAML.

Docker Registries

CONCEPT 03
3
What is a Docker Registry?

A Docker Registry is a storage and distribution system for Docker images. Just like GitHub stores code, a registry stores container images. When you run docker pull nginx, Docker downloads the nginx image from the Docker Hub registry. You can also push your own images so teammates, CI/CD pipelines, or cloud servers can pull and run them.

💡 AI Engineer Use Case You build a Docker image containing your trained model weights + inference server. Push it to AWS ECR or GitHub Container Registry. Your production GPU cluster pulls and runs it — guaranteed same environment every time.

Registry Ecosystem

🐳 Docker HubDefault public registry. Millions of official images. Free tier available. hub.docker.com
☁️ AWS ECRAmazon Elastic Container Registry. Private, scales with AWS. Integrates natively with ECS, EKS, Lambda.
🔵 GCP Artifact RegistryGoogle's registry. Works with GKE and Cloud Run. Replaced the older GCR.
🔷 Azure ACRAzure Container Registry. Geo-replicated, integrates with AKS and Azure Pipelines.
🐙 GitHub GHCRGitHub Container Registry. Free for public images. Stores images alongside your source code.
🏠 Self-HostedRun your own with the official registry:2 image. Full control, on-premises compliance.

Image Naming Convention

# Full image reference format:
[registry-host]/[namespace]/[repository]:[tag]

# Docker Hub (default registry, can omit host):
docker pull nginx:latest               # → docker.io/library/nginx:latest
docker pull username/myapp:v2.1        # → docker.io/username/myapp:v2.1

# AWS ECR:
docker pull 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:prod

# GitHub Container Registry:
docker pull ghcr.io/myorg/inference-server:sha-abc123
🏷️ Tagging Strategy Use semantic tags: :latest (unstable), :v1.0.0 (stable release), :sha-abc123 (immutable, tied to git commit). For AI models, tag with the model version too: :llama3-v1.2.
🔐 Private Registry Auth Use docker login before pushing to private registries. In CI/CD, inject credentials as environment variables. Never bake credentials into a Dockerfile.

Docker Networks — Advanced Concepts

CONCEPT 01 · EXTENDED

Container-to-Container Communication

On a user-defined bridge network, containers can reach each other using their service/container name as hostname. The embedded DNS server resolves names automatically. On the default bridge (docker0), you must use IP addresses or --link (deprecated).

Network Inspection & Debugging

When troubleshooting connectivity between containers, the most useful workflow is: inspect the network to see which containers are attached, then exec into a container and test connectivity with ping or curl.

# Create a custom network
docker network create --driver bridge my-ai-net

# Connect running container to a network (hot-attach)
docker network connect my-ai-net inference-server

# Inspect network — see all connected containers + IPs
docker network inspect my-ai-net

# Test connectivity from inside a container
docker exec -it myapp ping db
docker exec -it myapp curl http://api-server:8080/health

Port Publishing Deep Dive

Port publishing with -p exposes container ports to the host and beyond. The format is always HOST_PORT:CONTAINER_PORT.

# Bind to all host interfaces
docker run -p 8080:80 nginx

# Bind to specific IP only (secure)
docker run -p 127.0.0.1:8080:80 nginx

# Random host port
docker run -p 80 nginx

# UDP port
docker run -p 5353:5353/udp dns-server
⚠️ Security Tip Binding to 0.0.0.0 (default) exposes the port on ALL network interfaces, including public ones. In development, use 127.0.0.1:HOST:CONTAINER to limit access to localhost only.
🔍 View Port Mappings Run docker port <container> to quickly see all published port mappings for a running container.

Compose Advanced & Registry Workflows

CONCEPTS 02–03 · EXTENDED

Compose Health Checks & Profiles

Health checks let Compose wait until a service is truly ready (not just started) before starting dependent services.

db:
  image: postgres:15
  healthcheck:
    test: ["CMD", "pg_isready"]
    interval: 10s
    timeout: 5s
    retries: 5
api:
  depends_on:
    db:
      condition: service_healthy

Profiles let you define optional services activated only when needed (e.g., only start Prometheus in monitoring mode).

prometheus:
  image: prom/prometheus
  profiles:
    - "monitoring"

# Start with monitoring profile:
docker compose \
  --profile monitoring up

Scaling Services with Compose

# Scale the api service to 3 replicas
docker compose up --scale api=3

# In compose file, define resource limits
api:
  image: myapp:latest
  deploy:
    replicas: 3
    resources:
      limits:
        cpus: '0.5'
        memory: 512M

Full Registry Workflow — Push to Docker Hub

# Step 1: Build image with a proper tag
docker build -t username/inference-server:v1.0 .

# Step 2: Login to Docker Hub
docker login                              # prompts username + password/PAT

# Step 3: Push image to registry
docker push username/inference-server:v1.0

# Step 4: Also tag as latest
docker tag username/inference-server:v1.0 username/inference-server:latest
docker push username/inference-server:latest

# Step 5: On another machine — pull and run
docker pull username/inference-server:v1.0
docker run -d -p 8000:8000 username/inference-server:v1.0

Self-Hosted Registry with docker-compose.yml

version: "3.9"
services:
  registry:
    image: registry:2
    ports:
      - "5000:5000"
    volumes:
      - registry-data:/var/lib/registry
    environment:
      - REGISTRY_STORAGE_FILESYSTEM_ROOTDIRECTORY=/var/lib/registry
volumes:
  registry-data:

# Push to self-hosted registry:
docker tag myapp:latest localhost:5000/myapp:latest
docker push localhost:5000/myapp:latest
🔍 Inspecting Images in a Registry Use docker manifest inspect to check an image's architecture, layers, and digest without pulling it. Use docker image ls with filters to manage local image cache.

Command Reference — Networks & Compose

QUICK REFERENCE

Docker Network Commands

Command Topic What it does
docker network ls Network Lists all networks on the Docker host — shows name, driver, and scope.
docker network create <name> Network Creates a new user-defined bridge network. Containers on this network can communicate by name.
docker network create --driver overlay <name> Network Creates an overlay network for multi-host Docker Swarm communication using VXLAN tunneling.
docker network inspect <name> Network Shows detailed JSON info about a network: subnet, gateway, connected containers, and their IPs.
docker network connect <net> <container> Network Attaches a running container to an existing network. Container immediately gets a new IP on that subnet.
docker network disconnect <net> <container> Network Detaches a container from a network. Useful for isolating a container mid-run for debugging.
docker network rm <name> Network Removes a network. Fails if any containers are still connected. Disconnect all first.
docker network prune Network Removes all networks not currently used by any container. Great for cleaning up after testing.
docker run --network=host <image> Network Runs container using the host's network stack directly. No port mapping needed but no isolation.
docker run --network=none <image> Network Runs container with no network interfaces — completely isolated, useful for offline batch processing.
docker run -p 8080:80 <image> Network Maps host port 8080 to container port 80. External traffic hitting host:8080 is forwarded to the container.
docker port <container> Network Shows all port mappings for a running container — quick way to verify which ports are published.
docker run --network <name> <image> Network Starts a container and immediately attaches it to the specified custom network.

Docker Compose Commands

Command Topic What it does
docker compose up Compose Builds images (if needed), creates containers, networks, and volumes, then starts all services defined in the YAML.
docker compose up -d Compose Same as above but detached — runs in background so your terminal is free. Most common in production.
docker compose up --build Compose Forces rebuild of images before starting containers, even if Docker thinks the image is up-to-date.
docker compose down Compose Stops and removes all containers, networks created by Compose. Volumes are preserved by default.
docker compose down -v Compose Stops and removes containers AND volumes. Use for a clean slate — deletes database data too.
docker compose ps Compose Lists all containers in the current Compose project — shows name, state, and port bindings.
docker compose logs -f Compose Streams live logs from all services. Add a service name to filter: docker compose logs -f api.
docker compose exec <svc> bash Compose Opens an interactive shell inside a running service container — equivalent to docker exec for Compose.
docker compose build Compose Builds (or rebuilds) images defined with a build: key in the Compose file without starting containers.
docker compose pull Compose Pulls the latest version of all images specified in the Compose file from their respective registries.
docker compose restart <svc> Compose Restarts a specific service without recreating it. Useful for applying config changes that don't need a rebuild.
docker compose scale api=3 Compose Scales the api service to 3 running instances. Requires a load balancer to distribute traffic between replicas.
docker compose config Compose Validates and shows the merged effective configuration after resolving env vars and overrides. Great for debugging.
docker compose -f custom.yml up Compose Uses a custom-named Compose file instead of the default docker-compose.yml.

Command Reference — Registries & Summary

QUICK REFERENCE

Docker Registry Commands

Command Topic What it does
docker login Registry Authenticates with Docker Hub (or another registry if a hostname is given). Saves credentials in ~/.docker/config.json.
docker login <registry-url> Registry Logs into a private registry like AWS ECR, GHCR, or a self-hosted registry at the given URL.
docker logout Registry Removes stored credentials for a registry from your Docker config. Use in shared/CI environments after pushing.
docker pull <image>:<tag> Registry Downloads an image and all its layers from a registry to local cache. If the tag is omitted, pulls :latest.
docker push <image>:<tag> Registry Uploads a locally built/tagged image to the registry. Only uploads layers not already present (delta push).
docker tag <src> <dst> Registry Creates a new name/tag pointing to the same image layers. Used to rename or re-tag images before pushing to a registry.
docker image ls Registry Lists all locally cached images with their repository, tag, image ID, and size. Does not show dangling images by default.
docker image ls -a Registry Shows ALL images including intermediate layers and dangling images (untagged images from old builds).
docker image rm <image> Registry Removes a local image from the cache. Fails if a container (even stopped) is using it. Use -f to force.
docker image prune Registry Deletes all dangling (untagged) images to free up disk space. Add -a to remove all unused images.
docker image inspect <image> Registry Returns detailed JSON: image ID, layers, architecture, OS, environment variables, exposed ports, and more.
docker image history <image> Registry Shows the build history — each Dockerfile instruction that created a layer, its size, and timestamp.
docker search <term> Registry Searches Docker Hub for public images matching a term. Shows name, description, star count, and official flag.
docker manifest inspect <image> Registry Fetches the manifest from the registry without pulling — shows OS, architecture, digest, and layer info.
docker save -o file.tar <image> Registry Exports an image as a tarball to a file — useful for air-gapped environments with no internet registry access.
docker load -i file.tar Registry Loads an image from a tarball created by docker save. Restores image with original name and tag.

Key Concepts Summary

🌐 Networks
Bridge = default isolated · Host = shared stack · Overlay = multi-host · User-defined networks get DNS resolution by container name. Always use custom networks over default docker0.
📦 Docker Compose
Define multi-container stacks in YAML. Services communicate by service name. Use healthchecks for readiness. Profiles for optional services. Volumes for data persistence across restarts.
🗂️ Registries
Login → build → tag → push → pull → run. Tag images semantically (v1.0.0, sha-abc). Use private registries (ECR, GHCR, ACR) for production AI workloads. Never bake credentials into images.
