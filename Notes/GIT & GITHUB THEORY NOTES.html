Version Control
DEFINITION · WHY WE NEED IT · TYPES
Page 3
Definition

Version Control (also known as Source Control or Revision Control) is a system that records changes to one or more files over time so that you can recall specific versions later. It maintains a complete, timestamped history of every modification, enabling teams to collaborate without overwriting each other's work, revert to any prior state, and understand exactly what changed, when, and by whom.

Why Do We Need Version Control?

🔄 History & Rollback

Every change is timestamped and attributed to an author. If a bug is introduced, you can rewind the entire project to any earlier working state — even weeks or months back — without losing other work. For AI engineers, this means reverting a bad model architecture change instantly.

🤝 Team Collaboration

Multiple developers can work on different features simultaneously. Changes from each person are tracked independently and merged together, preventing one person's work from silently overwriting another's. Essential in data science teams where multiple experiments run in parallel.

🔬 Experimentation Safety

Developers can create isolated environments (branches) to try new ideas, test model architectures, or refactor pipelines — without risking the stability of the main codebase. If the experiment fails, it is simply discarded with zero consequence.

📋 Accountability & Audit

Every commit contains metadata — who made the change, when, and why (through the commit message). This creates an auditable trail critical for regulated industries and open-source projects. In AI, it lets you trace which code change caused a model accuracy drop.

Types of Version Control Systems

Evolution of Version Control Systems

Type 1 Local VCS

Tracks file changes in a simple local database on a single machine only. The earliest form of version control. RCS (Revision Control System) is the classic example. No network required, but absolutely no team collaboration possible. If your hard drive fails, all history is lost.

Type 2 Centralised VCS

A single server holds all versioned files; all clients check out from this central place. Examples: SVN (Subversion), CVS, Perforce. Enables collaboration, but has a single point of failure — if the server goes down, nobody can commit or view history.

Type 3 Distributed VCS

Every developer has a complete clone of the entire repository including its full history. Examples: Git, Mercurial. You can commit offline; any copy can serve as the authoritative server. This is the modern standard and what Git uses.

What is Git?

Git is a free, open-source Distributed Version Control System (DVCS) created by Linus Torvalds in April 2005 originally to manage the Linux kernel source code. Git operates entirely on your local machine — it requires no internet connection to take snapshots, view history, branch, or commit. It stores data as a series of snapshots (not diffs), uses SHA-1 hashing for data integrity, and is designed for speed and support for non-linear distributed workflows.

What is GitHub?

GitHub is a cloud-based web hosting platform for Git repositories, acquired by Microsoft in 2018. GitHub adds collaboration tooling on top of Git — Pull Requests, Issues, Projects, GitHub Actions (CI/CD), and code review features. GitHub is not Git; it is the most popular remote server for storing and sharing Git repositories over the internet. Other alternatives include GitLab, Bitbucket, and Azure DevOps.

What is a Repository?

Definition

A Repository (commonly called a "repo") is the core data structure of Git — a special directory that stores the complete history of all changes ever made to a project, along with metadata such as author information, timestamps, branch pointers, and tags. The repository lives inside a hidden .git folder at the root of your project directory. Everything Git knows about your project — every commit, every branch, every object — lives inside this single folder.

Types of Repositories

💻 Local Repository

A repository that resides on your own computer inside the .git directory. All Git operations — committing, branching, viewing history, staging — happen against the local repository. It is completely independent and works without any network connection. Each developer has their own private local repository for their working copy of the project.

☁️ Remote Repository

A repository hosted on a server or cloud platform (like GitHub). It acts as the central authoritative source of truth that all team members synchronise with. You push your local commits to it and pull others' changes from it. A remote repository typically does not have a working directory — it stores only the bare Git data objects.

🔒 Private Repository

A repository whose access is restricted to specific invited collaborators only. The code, commit history, and issues are invisible to the public. Used for proprietary software, client projects, internal tools, and AI model codebases where intellectual property must be protected. GitHub requires a paid plan for unlimited private collaborators in organisations.

🌐 Public Repository

A repository visible to anyone on the internet. Anyone can view the code, clone it, and (with the owner's permission) submit Pull Requests. Public repositories are the foundation of open-source software. Most popular AI/ML frameworks — TensorFlow, PyTorch, Hugging Face Transformers — are public repositories on GitHub.

🍴 Forked Repository

A personal copy of someone else's repository created on your own GitHub account. Forking creates an independent duplicate that you can freely modify without affecting the original project. When your changes are ready, you can propose them back to the original repository via a Pull Request. Essential for open-source contributions.

📜 Bare Repository

A repository that contains only the version history (the contents of the .git folder) with no working directory for editing files. Bare repositories are used exclusively as server-side or remote repositories — you cannot directly edit files in them; you can only push to and fetch from them. GitHub stores all repos as bare repositories on its servers.

Complete Git Architecture — Four Areas, Their Components and Connections

Area 1 Working Directory

Your actual project folder on disk — the files you see and edit in your code editor. Git classifies these as tracked (files Git knows from a prior commit) or untracked (new files Git has never seen). Modified tracked files appear as "modified"; new files appear as "untracked" in git status.

Area 2 Staging Area (Index)

A preparation zone — a temporary holding area between your working directory and repository. When you run git add, Git takes a snapshot of that file and places it here. You can stage some files and leave others unstaged, giving fine-grained control over what each commit contains.

Area 3 Local Repository (.git)

The hidden .git folder that stores the complete object database — all commits, branches, tags, and configuration. When you git commit, Git permanently records the staged snapshot here using SHA-1 hashed objects (blobs for file content, trees for directory structure, commit objects).

Area 4 Remote Repository (GitHub)

A version of your repository hosted on a server. When you git push, your local commits are uploaded to the remote. When you git pull, remote changes are downloaded and merged locally. The remote is the shared source of truth for the entire team.

What is a Commit?

A commit is a permanent, immutable snapshot of your staged changes stored in the Git repository. It is the fundamental unit of history in Git. Each commit is uniquely identified by a SHA-1 hash (a 40-character hexadecimal string) and captures the complete state of all tracked files at that exact moment — not just the diff. Once created, a commit object can never be changed; it can only be referenced or built upon. This immutability guarantees the integrity of your project history.

Commit Types — Conventional Commits Standard

The Conventional Commits specification is an industry-wide standard for writing structured commit messages. It enables automatic changelog generation, semantic versioning, and greatly improved readability across teams. Format: <type>(optional scope): <short description>

feat

Feature Commit

Introduces a new feature or capability to the codebase. The most common type in active development. Example: feat: add BERT fine-tuning pipeline. Triggers a MINOR semantic version bump (1.0.0 → 1.1.0). In AI projects, this could mean adding a new model architecture or training strategy.

fix

Bug Fix Commit

Patches a bug or defect in the existing code. Example: fix: resolve NaN loss during training. Triggers a PATCH version bump (1.0.0 → 1.0.1). Directly corresponds to something a user noticed as broken that is now repaired.

docs

Documentation Commit

Changes that affect only documentation — README files, docstrings, API references, wikis. Example: docs: update model training instructions. Does not change any source code logic. Does not trigger a version bump.

refactor

Refactor Commit

A change that neither fixes a bug nor adds a feature — improves code structure, readability, or performance without altering external behaviour. Example: refactor: extract model layers into separate module. Very common in AI code cleanup.

test

Test Commit

Adding or correcting automated tests without modifying production code. Example: test: add unit tests for data preprocessing pipeline. Critical in AI pipelines to verify that data transformations, feature engineering steps, and model evaluations produce correct outputs.

chore

Chore Commit

Maintenance tasks that don't modify source or test files — updating build scripts, package dependencies, CI configuration. Example: chore: upgrade torch from 2.2.0 to 2.3.0. The most common type when updating ML library versions in AI projects.

feat!

Breaking Change Commit

Introduces an incompatible API or behaviour change. Indicated by appending ! after the type or adding BREAKING CHANGE: in the commit footer. Example: feat!: redesign model configuration schema. Triggers a MAJOR version bump (1.0.0 → 2.0.0).

Branching
CONCEPT · HEAD POINTER · STRATEGY · DIAGRAMS
Page 7
What is a Branch?

A branch in Git is a lightweight, movable pointer to a specific commit. Internally it is nothing more than a 41-byte file containing a SHA-1 hash. When you create a branch, Git creates this tiny pointer — it does not copy any files or commit objects. This makes branch creation in Git nearly instantaneous and storage-free, regardless of project size. A branch represents an independent line of development, allowing you to work on a feature, bug fix, or ML experiment in complete isolation without disturbing the stable codebase.

The HEAD Pointer

HEAD is Git's special pointer that always answers the question: "Where am I right now?" HEAD points to the currently checked-out branch (which in turn points to the latest commit on that branch). When you switch branches with git checkout, HEAD moves to point to the new branch. In a detached HEAD state, HEAD points directly to a commit rather than a branch — this happens when you check out a specific commit hash or tag.

Git Flow — The Standard Branching Strategy

main

Main / Master Branch

Production-ready, stable code only. Every commit on main represents a deployed release. No direct commits are ever made here — only merges from release or hotfix branches. Deploying AI models from main is standard practice.

develop

Develop Branch

The primary integration branch for all completed features. Feature branches merge here first. Acts as the staging area — code here should always be in a working (if not necessarily release-ready) state.

feature/

Feature Branches

Short-lived branches created from develop. Each branch represents exactly one task or user story. Naming convention: feature/add-attention-layer, feature/data-augmentation.

hotfix/

Hotfix Branches

Branched directly from main to address critical production bugs immediately. After fixing, merged back into both main AND develop so the fix is not lost in future releases.

release/

Release Branches

Created from develop when a release is ready for QA. Only bug fixes, documentation updates, and version bumps happen here. Merged into both main and develop when complete.

experiment/

Experiment Branches (AI-Specific)

In AI projects, engineers create experiment/ branches to test new architectures, hyperparameters, or training datasets. These branches may never merge — they exist purely to capture a reproducible code state tied to an ML experiment run.

What is Merging?

Merging is the process of integrating the history of one branch into another. When you merge a feature branch back into main, Git locates the common ancestor commit (the point where both branches diverged), then combines the changes made on each branch since that point. Git preserves the full history of both branches, creating an accurate record of how development actually happened.

What is Rebasing?

Rebasing re-applies your branch's commits on top of another branch's latest commit, creating a clean, linear history. Git takes each commit from your feature branch, temporarily removes them, fast-forwards your branch to the tip of the target branch, then replays each commit one by one as brand-new commits with new SHA hashes. The result looks as though you wrote all your changes after the latest main commit — no merge commit, no divergence in the log. Because it rewrites commit hashes, rebase must never be used on shared or public branches.

Squash Merging

What is Squash Merging?

Squash merging combines all commits from a feature branch into a single new commit that is placed on the target branch. Instead of preserving every small, intermediate commit (like "WIP", "fix typo", "try again"), squash merging presents the entire feature as one clean, cohesive commit. This results in a very clean, readable main branch history. The original feature branch's individual commits are not preserved in the main history. This is the preferred merge strategy on many GitHub-hosted AI projects using Pull Requests.

The Golden Rule of Rebasing: Never rebase a branch that other people are working on. Because rebase creates brand-new commit objects (with different SHA hashes), anyone who based their work on the original commits will have severe conflicts. Only rebase your local, private feature branches before pushing them for the first time.

Complete Remote Operation Flow — Local Machine to GitHub
git push

Push uploads your local commits to the corresponding branch on the remote repository. Git transfers all new commit objects that the remote doesn't yet have. If the remote has commits you lack, Git rejects the push — you must pull first to incorporate their work before yours can be added on top.

git pull

Pull is a combination of two operations: git fetch followed immediately by git merge. It downloads all changes from the remote and integrates them into your current branch in one step. This is the most common way to update your local branch with the latest teammates' work.

git fetch

Fetch downloads all changes from the remote into your local repository but does not modify your working directory or current branch at all. It only updates the remote-tracking references (like origin/main). This lets you inspect what has changed on the remote before deciding whether and how to merge.

git clone

Clone creates a complete local copy of a remote repository — including all commits, branches, and tags. It automatically sets up the remote as origin and checks out the default branch. Cloning is how you first get any project onto your machine, including AI/ML repositories like PyTorch or Hugging Face models.

Merge Conflicts

What is a Merge Conflict?

A merge conflict occurs when two branches have made different changes to the exact same lines of the same file, and Git cannot automatically decide which version is correct. Git pauses the merge and marks the conflicting sections inside the file using conflict markers. The developer must manually open the file, examine both versions, decide what the correct final version should be (often combining elements of both), remove the conflict markers, and then stage the resolved file to complete the merge.

How Conflict Markers Look Inside a File

Remote-Tracking Branches

When you clone or fetch a repository, Git creates remote-tracking branches such as origin/main and origin/develop. These are local read-only references that reflect the state of the remote branches at the time of your last fetch or pull. They serve as bookmarks so Git can calculate what has diverged between your local branches and the remote. You cannot commit directly to them.

✓
Best Practice — Pull Before You Push: Always run git pull (or git fetch and review) before pushing, to avoid unnecessary conflicts. In AI teams working on shared training scripts, config files, or requirements.txt, this single habit prevents the majority of merge conflicts.

📦 Git Stash — Temporarily Save Uncommitted Work

What is Git Stash?

Git stash temporarily saves your uncommitted changes (both staged and unstaged) onto a private stack, then restores your working directory to a clean state — as if you hadn't changed anything yet. You can later re-apply those saved changes and continue where you left off. Think of it like putting your current work in a drawer so you can deal with something urgent, then coming back to that drawer afterwards.

When Do You Use It?

🚨 Urgent Context Switch

You're halfway through editing a model's training loop when your team lead asks you to urgently fix a bug on the main branch. You can't commit half-done work, so you stash it, switch branches to fix the bug, then come back and pop your stash to resume exactly where you left off.

📋 Reviewing a Colleague's PR

You need to checkout a teammate's branch to review their code, but you have local uncommitted changes that would conflict. Stash your work, checkout their branch, review it, then return to your branch and pop your stash back out.

Key Rules to Remember

  • Stash is a stack — you can have multiple stashes saved at once. Each is identified by an index: stash@{0} is the most recent, stash@{1} is the one before that, and so on.
  • Stashes are local only — they live on your machine and are never pushed to a remote repository. Teammates cannot see your stash.
  • Untracked files are excluded by default — if you've created a brand new file that you haven't run git add on yet, it won't be stashed unless you explicitly add the --include-untracked flag.
  • Stash does not replace commits — stash is for temporary, in-progress work only. When you're done with a feature, you still need to commit it properly.

🅹 Git Cherry-Pick — Apply a Single Specific Commit

What is Cherry-Pick?

Cherry-pick lets you take one specific commit from any branch and apply its changes onto your current branch — without merging the entire source branch. Git creates a brand new commit on your branch containing the same change, but it gets a different SHA hash because it's technically a new commit object in a different location in history.

✅ Classic Use Case

A critical bug fix was committed to the develop branch. You need that fix on main immediately — but develop also has half-finished features you don't want to release. Cherry-pick lets you copy just that one bug fix commit across to main, leaving the rest of develop untouched.

⚠️ Important Caution

Cherry-picking creates a duplicate commit with a different SHA. If you later merge the original branch, Git may see the same change twice and create a conflict. Use cherry-pick sparingly — it's a targeted tool for emergencies, not a substitute for proper branching and merging.

🏷️ Git Tags — Permanent Version Markers

What is a Tag?

A tag is a permanent, named label that points to one specific commit forever. Unlike a branch (which moves forward with every new commit), a tag never moves — it always points to the same commit it was created on. Tags are used to mark release versions such as v1.0.0, v2.3.1. In AI engineering, tags mark the exact code commit that corresponds to a specific trained model deployment.

Lightweight Tag

Simply a named pointer to a commit — nothing more. No extra information is stored. Think of it like a sticky note on a commit. Created with: git tag v1.0.0. Useful for quick personal bookmarks but not recommended for official releases.

Annotated Tag (Preferred)

A full Git object that stores: the tagger's name, email, the date, and a tagging message. Can also be GPG-signed for cryptographic verification. Created with: git tag -a v1.0.0 -m "Release v1.0.0". Always use annotated tags for official releases.

⚠️
Tags are not pushed automatically. A regular git push does not upload your tags to the remote. You must explicitly run git push origin --tags to share them with the team.

🚫 .gitignore — Telling Git What to Ignore

What is .gitignore?

The .gitignore file is a plain-text file at the root of your repository that lists patterns of files and folders Git should completely ignore. Any path matching a pattern in this file will never appear in git status, will never be staged, and will never be committed — Git treats it as if it doesn't exist. The .gitignore file itself should be committed to the repository so the whole team shares the same rules.

📝 Git Log — Reading Your Project's History

What is git log?

git log is the command you use to read and explore your repository's commit history. It starts from HEAD (your current position) and walks backwards through every commit that led to this point, displaying information about each one: its SHA hash, who made it, when, and the commit message. In AI engineering, git log is how you trace exactly which code change caused a model accuracy drop — you can identify the exact commit, inspect it, and revert it if needed.

Useful Ways to Use git log

📄 git log --oneline

Shows one commit per line with a shortened SHA and just the subject line of the commit message. The most readable format for quickly scanning recent history. Use this daily to see what you and your team have committed.

🌳 git log --oneline --graph

Adds an ASCII art diagram showing the branch and merge structure visually. Lets you see at a glance how feature branches diverged and where they were merged back in. Essential for understanding complex multi-branch histories.

👤 git log --author="Name"

Filters history to show only commits made by a specific person. Useful when you want to review what a particular teammate contributed, or to audit your own commits before a PR review.

📅 git log --since="2 weeks ago"

Filters history to commits after a specific date. Combine with --until to set a date range. Excellent for sprint retrospectives — see everything the team changed in the past two weeks.

🔍 Git Diff — Seeing Exactly What Changed

What is git diff?

git diff shows you the exact line-by-line differences between two states of your code. Lines prefixed with + (shown in green) are additions. Lines prefixed with - (shown in red) are deletions. Git diff is how you answer the question: "What exactly did I change since the last commit?" In AI engineering, running git diff before committing is a professional habit that catches accidental debugging prints, hardcoded paths, and incomplete edits before they reach the repository.

Three Forms of git diff You Must Know

git diff

Unstaged changes only. Shows the difference between your current working files and the last commit. These are changes you've made but haven't run git add on yet. This is the default and most common use.

git diff --staged

Staged changes only. Shows the difference between what you've staged with git add and the last commit. Run this immediately before git commit to do a final review of exactly what you are about to commit.

git diff A..B

Between any two points. Compare two branch names, two commit SHAs, or two tags. For example, git diff main..feature/lstm shows all differences between the main branch and your feature branch — exactly what a PR would add.

git diff — Where Each Form Reads From
💡
Professional Habit: Before every git commit, always run git diff --staged first. This gives you one final chance to review every line you're about to save permanently. Senior engineers never commit without checking their diff — it prevents committing debug prints, test API keys, and incomplete changes.

🔍 Putting It Together — Debugging a Model Regression with Git

Scenario

Your model's validation accuracy dropped from 92% to 84% after last week's work. You need to find which commit caused it. Use git log --oneline to list recent commits, identify the suspicious range, then use git diff SHA1..SHA2 to see exactly what changed between two commits. Once found, git revert that commit safely.

git bisect (Advanced)

git bisect automates this regression hunt. You tell Git a "good" commit (accuracy was fine) and a "bad" commit (accuracy is broken). Git then does a binary search through history — checking out commits halfway in between and asking you to test each one — until it pinpoints the exact commit that introduced the regression.

Pull Requests (PRs)

Definition

A Pull Request is a GitHub-level feature that lets you formally propose merging your branch into another branch (usually main or develop). It opens a dedicated discussion space where teammates can review every changed line of code, leave inline comments, suggest improvements, and ultimately approve or request changes before the merge is permitted. PRs are the primary collaboration mechanism on GitHub and the standard way AI teams review model code, training scripts, and configuration changes before they reach production.

Pull Request Lifecycle — From Branch to Merged Code

Code Review on GitHub

Code review is the structured process of having teammates examine proposed code changes before they are merged. On GitHub, reviews happen inside a Pull Request and include three possible outcomes: Comment (general feedback without explicit approval), Approve (the reviewer signs off and approves the merge), or Request Changes (the reviewer blocks the merge and requires specific fixes before re-reviewing).

👁️ What Reviewers Check

Logic correctness, code readability, naming conventions, missing tests, security vulnerabilities, performance concerns, and adherence to the team's coding standards. In AI, reviewers also check for data leakage, incorrect loss functions, and hardcoded values.

📋 Branch Protection Rules

GitHub allows repository admins to require a minimum number of PR approvals before any merge is permitted. This enforces the code review process at the infrastructure level. Configured under Repository Settings → Branches.

✨ Draft PRs

A Draft Pull Request signals that work is in progress and not yet ready for review. It appears in the PR list but cannot be accidentally merged. Used by AI engineers to get early feedback on a new model architecture before it's complete.

GitHub Actions — CI/CD for AI Engineers

What is GitHub Actions?

GitHub Actions is GitHub's built-in CI/CD (Continuous Integration / Continuous Deployment) platform that automates workflows directly from your repository. A workflow is defined as a YAML file placed inside .github/workflows/ and is triggered by specific GitHub events such as a push to main, a new Pull Request, a scheduled time, or a manual trigger. Each workflow consists of one or more jobs that run in sequence or parallel, and each job consists of steps that execute commands or pre-built community Actions.

🔄 Continuous Integration

Automatically run test suites, linters (flake8, black), and type checkers (mypy) on every push or PR. This catches regressions in data preprocessing or training logic before they reach main. The PR is blocked from merging if any check fails.

🧪 Model Validation

On each PR, automatically run a lightweight model evaluation job — train on a small subset, evaluate on a validation split, assert that key metrics (accuracy, F1, loss) remain above a threshold. Alerts the team immediately when a code change breaks model behaviour.

🚀 Deployment Automation

When a version tag (v1.2.0) is pushed, Actions can automatically build a Docker container with the model, push it to a registry (DockerHub, ECR), and deploy the updated inference API to a cloud environment (AWS, GCP, Azure) without any manual steps.

GitHub Issues

Issues are GitHub's built-in task and bug tracking system. Every repository has an Issues tab where team members can create tickets describing bugs, feature requests, questions, or tasks. Issues can be assigned to specific team members, labelled (e.g., bug, enhancement, model-performance), linked to milestones, and referenced from commit messages or PR descriptions. When a PR contains Fixes #42 in its description, GitHub automatically closes Issue #42 when that PR is merged.

💡
Issues + PR + Actions = Complete Workflow: A junior AI engineer's full professional workflow is: create an Issue describing the task → create a feature branch → write code → push → Actions run tests automatically → open a PR referencing the Issue → teammates review → merge → Issue auto-closes → Actions deploy. This is the complete, industry-standard loop.

Git Internals — How Git Stores Data

Page 12
Git's Object Database

Underneath Git's commands lies a simple but powerful content-addressable object store. Everything Git stores — every version of every file, every directory structure, every commit — is saved as an object in the .git/objects/ directory. Each object is identified by its SHA-1 hash: a 40-character hexadecimal fingerprint computed from the object's content. If two files have identical content, they share one object. If a single byte changes, a completely different SHA-1 is produced — this is what makes Git's history tamper-proof and its deduplication efficient.

The Four Types of Git Objects

Git Object Hierarchy — Blob → Tree → Commit → Tag

📄 Blob Object

A blob (binary large object) stores the raw content of a single file version. It contains only the file's bytes — no filename, no permissions. Two files in different directories with identical content share the same blob object, saving space. One new blob is created every time a file's content changes.

📁 Tree Object

A tree represents a directory snapshot. It contains a list of entries, each mapping a filename or subdirectory name to a blob SHA (for files) or another tree SHA (for subdirectories), plus file permission modes. The root tree of a commit represents the complete project directory structure at that point in time.

📤 Commit Object

A commit object points to a root tree (the project snapshot), stores the parent commit SHA(s), records author and committer metadata (name, email, timestamp), and contains the commit message. The commit object is what you interact with via git log — it is the top-level entry point into a complete project snapshot.

SHA-1 and Data Integrity

How SHA-1 Protects Your History

Every Git object is identified by a SHA-1 hash computed from its entire content. This means that if anyone (or any disk error) changes even a single bit of any commit, file, or tree object, its SHA-1 changes completely. Since commits reference parent commits by their SHA-1, and trees reference blobs by their SHA-1, the entire history forms a cryptographically linked chain. Git verifies these hashes automatically — if data is corrupted, Git will detect it immediately. This is why Git history is considered tamper-evident and trustworthy for AI model reproducibility and compliance requirements. Note: Git is transitioning from SHA-1 to SHA-256 (via git init --object-format=sha256) for improved security.

Git Workflows & Collaboration Patterns

Page 13
Why Workflows Matter

A Git workflow is a team agreement on how branches are named, created, merged, and deleted. Without a defined workflow, teams quickly develop conflicting histories, accidental merges, and unclear deployment processes. Choosing the right workflow depends on team size, release frequency, and the nature of the project. For AI teams, workflow choice directly impacts how experiments are tracked, how models are versioned, and how quickly bug fixes reach production.

1. Git Flow

Git Flow — Full Branch Model for Structured Releases

Best for: Structured, versioned software with scheduled releases. Uses main (production), develop (integration), feature/, release/, and hotfix/ branches. Complex but very controlled. Good for enterprise AI product teams shipping model APIs on a quarterly release cycle.

2. GitHub Flow

GitHub Flow — Simple, Fast, Modern

GitHub Flow is a simplified, lightweight workflow designed around continuous deployment. It has just two rules: (1) anything on main is always deployable, and (2) to work on anything new, create a descriptively named branch from main, open a Pull Request, get it reviewed, merge it, and deploy immediately. There are no separate develop, release, or hotfix branches. This simplicity makes it ideal for teams deploying frequently — multiple times per day. Most modern SaaS AI products use GitHub Flow.

Best for: Teams doing continuous deployment. Clean, minimal branching model — perfect for web-based AI services that deploy after every merged PR.

3. Trunk-Based Development

Trunk-Based Development

In Trunk-Based Development, all developers commit small, frequent changes directly to a single shared branch called main (or trunk), several times a day. Short-lived feature branches lasting no more than 1-2 days are acceptable, but the emphasis is on integrating continuously to avoid long-lived divergences. This workflow requires robust automated testing (CI) to ensure main is always green. It is the workflow Google and Facebook use internally at massive scale. Increasingly popular in large AI labs with mature CI/CD pipelines.

4. Forking Workflow

Forking Workflow — Open-Source Standard

In the Forking Workflow, every contributor (especially external ones) has their own fork (personal server-side copy) of the official repository. Contributors push changes to their own fork, never to the official repo, and then open Pull Requests from their fork's branch to the official repository's branch. This isolates contributors from the official codebase entirely — maintainers can review and accept contributions without giving strangers push access. This is the standard workflow for all open-source projects on GitHub, including contributing to frameworks like LangChain, Hugging Face, or scikit-learn.

💡
Which workflow to use as a Junior AI Engineer? Most AI startups and product teams use GitHub Flow for its simplicity. Large regulated enterprises use Git Flow. Open-source contributions use the Forking Workflow. Understanding all four makes you adaptable to any team you join.

DVC — Data Version Control for AI Engineers

What is DVC?

DVC (Data Version Control) is an open-source tool that extends Git to handle the large binary files that AI/ML projects produce — datasets, trained model weights, feature stores, and evaluation results. Git is not designed for files over 100 MB (GitHub even blocks files over 100 MB). DVC solves this by storing only a small pointer file (.dvc file) in Git while the actual large file is pushed to remote storage (Amazon S3, Google Cloud Storage, Azure Blob, a shared NFS drive, etc.). This means your full experiment is reproducible: checking out any Git commit also restores the exact data and model state for that experiment.

📈 How DVC Works

DVC tracks large files via a content-addressable cache (similar to Git's object store). Running dvc add data.csv adds a data.csv.dvc pointer file to Git and caches the actual file locally, then pushes to remote storage with dvc push. Teammates run dvc pull to download it.

🔗 Git + DVC Together

Git tracks: code, configs, requirements, .dvc pointer files, dvc.yaml pipeline definitions. DVC tracks: actual data files, model binaries, experiment results. Together they provide full ML experiment reproducibility tied to a Git commit SHA.

🧪 DVC Pipelines

DVC can define ML pipelines as a DAG (directed acyclic graph) of stages in a dvc.yaml file. Each stage has inputs, outputs, and a command. Re-running dvc repro only re-executes stages whose dependencies have changed — similar to how make works for build systems.

Git Configuration

git config

Git has three levels of configuration: System (applies to all users on the machine, stored at /etc/gitconfig), Global (applies to all repositories for your user, stored at ~/.gitconfig), and Local (applies only to the current repository, stored at .git/config). Local overrides global, which overrides system. Before using Git for the first time, you must configure at minimum your name and email, as these are embedded in every commit you create.

👤 Identity Settings

Your name and email are embedded into every commit object. They are public in any public repository. Set them globally once: name maps to the "Author" field; email maps to your GitHub account for avatar display in commit history and blame views.

⚙️ Useful Global Settings

Configure your preferred text editor for commit messages (core.editor), default branch name for new repos (init.defaultBranch main), credential helper to avoid typing passwords repeatedly, and colour output (color.ui auto) for readable terminal output.

✏️ Writing Great Commit Messages

Why Commit Messages Matter

A commit message is a permanent note to your future self and your teammates explaining what you changed and why. When something breaks six months later, your team will use git log and git blame to find the relevant commit. A message like "fix stuff" is useless. A message like "fix: resolve NaN loss caused by zero-division in batch normalisation layer" is invaluable. Good commit messages transform your project history from a confusing list of timestamps into a readable, professional record of decisions.

The Two Parts of a Commit Message

Part 1 — Subject Line (Required)

The first line of your commit message. This is what appears in git log --oneline, on GitHub's commit list, and in email notifications. Keep it short, clear, and imperative. It's the headline of your change.

Part 2 — Body (Optional but Valuable)

Written after a blank line below the subject. The body explains what the problem was and why this approach was chosen — not how (the code already shows that). Incredibly valuable for complex changes that need context.

The 7 Rules of a Great Commit Message

Rule 1

Limit subject to 50 characters

Forces clarity. Displays fully in all Git tools without wrapping or truncating. Count before you commit.

Rule 2

Capitalise the subject line

Standard writing convention. "Add dropout layer" not "add dropout layer". Reads more naturally in log views.

Rule 3

No period at the end of the subject

Subject lines are titles, not sentences. "Fix training loop" — not "Fix training loop." The period wastes one of your 50 characters.

Rule 4

Use the imperative mood

Write as if giving a command: "Add LSTM layer" — not "Added" or "Adding". Git's own messages do this ("Merge branch..."). Think: "This commit will [your subject line]."

Rule 5

Wrap body at 72 characters

Git doesn't auto-wrap text. The 72-character limit is the standard terminal and diff-view width. Manually add line breaks in the body.

Rule 6

Body explains WHAT & WHY, not HOW

The code shows how. The commit body explains what problem existed and why this particular solution was chosen. Context that will be invaluable 6 months later.

Rule 7

Blank line between subject and body

Tools like git log --oneline and GitHub use this blank line to know where the subject ends. Without it, everything runs together as one paragraph.

✅ Example of a Perfect Commit Message

Subject: feat(training): Add cosine annealing LR scheduler

Body: The model was hitting loss plateaus after epoch 15 with the constant learning rate of 0.001. Cosine annealing was chosen over step decay because it provides smoother convergence on the validation set and avoids the sharp loss jumps caused by sudden LR drops. Tested over 5 runs — validation accuracy improved from 88.2% to 91.4%.

✅ Best Practices for Junior AI Engineers

These are the habits that separate a professional AI engineer from a beginner. Follow these consistently from day one.

📤 Commit Early & Often

Each commit should represent one single logical change. Small, focused commits are easier to review, understand, and revert. Avoid large monolithic commits that touch 20 files at once — they are impossible to review meaningfully and catastrophic to roll back.

✍️ Write Meaningful Commit Messages

Follow the Conventional Commits standard and the 7 rules from the previous page. "fix: resolve NaN loss in batch normalisation" is infinitely more useful than "fix stuff". Your commit history is your project's permanent scientific record.

🚫 Never Commit to Main Directly

Even for quick solo fixes, always work on a feature branch and open a Pull Request. This keeps main always deployable, forces you to review your own work, and builds the PR habit essential for team environments.

🔒 Keep .gitignore Updated From Day One

Set up your .gitignore before writing a single line of code. Preventing model weights, datasets, and API keys from entering Git history is far easier than attempting to remove them after they've been pushed to a remote.

💾 Use DVC for Large Files

Git is not designed for files over 100 MB. GitHub blocks files over 100 MB outright. Use DVC alongside Git to version datasets and trained model artefacts in cloud storage (S3, GCS, Azure Blob), keeping the Git repository lean and fast to clone.

📊 Pull Before You Push

Always run git pull to synchronise with the remote repository before you push your changes. This ensures you resolve any conflicts with your teammates' work locally before they reach the shared remote branch.

👀 Review Your Own Diff Before Committing

Run git diff --staged before every git commit. Read your own changes line by line. This catches debugging prints, hardcoded local paths, test API keys, and incomplete edits before they become permanent.

🏷️ Tag Every Model Release

Use annotated tags (git tag -a v1.0.0 -m "...") to mark each commit that corresponds to a deployed model version. This creates a permanent, reproducible link between your code state and your model's behaviour in production.

❌ Common Mistakes to Avoid

❌ Committing to main directly

Bypasses code review, breaks CI gates, and can push untested code to production. Always use a feature branch — even for tiny fixes.

❌ Huge, infrequent commits

Committing three days of work at once makes code review impossible and reverting extremely destructive. Break work into small logical steps and commit each one.

❌ Rebasing shared branches

Rebasing a branch that teammates have built their work on top of rewrites its history and causes catastrophic conflicts. Only rebase local, unshared branches.

❌ Committing large files

Model weights and datasets bloat the repository permanently. Once in history, large files cannot be cleanly removed without rewriting all history. Use DVC from the start.

❌ Committing .env / secrets

API keys in Git history are visible forever, even after deletion in a later commit. If this happens, rotate the credential immediately — removing it from history is not sufficient.

❌ Force-pushing to main

git push --force rewrites remote history and destroys teammates' work. One of the most destructive Git actions possible. Prevent it with branch protection rules.
