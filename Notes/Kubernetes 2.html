Kubernetes Level 201 — YAML Manifests
01
Concept #1

01.YAML Manifests — The K8s Config Language

📖 What Is a Manifest? A Kubernetes manifest is a YAML file that declares the desired state of a Kubernetes object. Instead of imperative commands like kubectl create deployment, you describe what you want in a file and apply it. This enables GitOps — your entire infrastructure is code, stored in git, reproducible and auditable.

The Four Required Fields Every Manifest Has

any-resource.yaml · The Four Root Fields
apiVersion: apps/v1 # Which K8s API group handles this resource kind: Deployment # What type of K8s object this is metadata: # Identity: name, namespace, labels name: my-rag-app namespace: production labels: app: rag-app version: v2 spec: # Desired state — differs per resource type replicas: 3

apiVersion Reference

Object TypeapiVersionNotes
Deployment, ReplicaSet, DaemonSet, StatefulSetapps/v1Core app workloads, stable
Pod, Service, ConfigMap, Secret, Namespace, PVCv1No group prefix — core objects
Job, CronJobbatch/v1Batch workloads
Ingress, NetworkPolicynetworking.k8s.io/v1Network routing & firewall rules
HorizontalPodAutoscalerautoscaling/v2Autoscaling objects
Role, ClusterRole, RoleBindingrbac.authorization.k8s.io/v1Access control

Labels vs Annotations

Labels — For Selecting & Filtering Key-value pairs used by Kubernetes to identify and group objects. Services use label selectors to find which Pods to route traffic to. Must match exactly between selector and Pod template. Keep them short and machine-readable.
Annotations — For Metadata Only Arbitrary metadata K8s doesn't use for selection. Use for: build number, git SHA, Prometheus scrape config, cert-manager directives. Values can be long strings. Example: prometheus.io/scrape: "true".

The kubectl apply Workflow

Terminal · apply-based workflow
$ kubectl apply -f deployment.yaml # Create or update from YAML $ kubectl apply -f ./k8s/ # Apply every .yaml in a directory $ kubectl apply -f ./k8s/ --dry-run=client # Validate without applying $ kubectl diff -f deployment.yaml # Show what WOULD change $ kubectl get deployment my-rag -o yaml # Export live resource as YAML
💡 Always Use apply, Not create kubectl create -f fails if the resource already exists. kubectl apply -f creates if absent, updates if present. Always use apply — it's safe to run repeatedly and is the foundation of GitOps.

Complete, annotated YAML for a production-ready RAG API Deployment paired with its Service. Every field explained.

rag-deployment.yaml · Fully Annotated

Kubernetes Level 203 — ConfigMaps & Secrets as YAML
03
Concept #3

03.ConfigMaps & Secrets as YAML

At Level 1 you created ConfigMaps and Secrets imperatively. In production you write them as YAML and commit them to git (with Secrets encrypted or managed by a secrets tool).

rag-config.yaml · ConfigMap + Secret
apiVersion: v1 kind: ConfigMap metadata: name: rag-config namespace: production data: MODEL_NAME: "gpt-4o" CHUNK_SIZE: "500" # Numbers must be quoted strings LOG_LEVEL: "INFO" app.config.yaml: | # Entire config FILE stored as a key retrieval: top_k: 5 score_threshold: 0.75 embedding: model: text-embedding-3-small --- apiVersion: v1 kind: Secret metadata: name: api-keys namespace: production type: Opaque # Generic secret type data: # Values MUST be base64 encoded OPENAI_KEY: c2stcHJvai0... # echo -n "sk-proj-..." | base64 PINECONE_KEY: cGMtdGVzdC4u

Mounting a ConfigMap as a File Inside a Container

Pod spec · Mounting ConfigMap as File
spec: volumes: - name: config-volume configMap: name: rag-config items: - key: app.config.yaml # Which key to mount path: config.yaml # Filename inside the container containers: - name: rag-api volumeMounts: - name: config-volume mountPath: /app/config # Container sees /app/config/config.yaml readOnly: true
🔑 ConfigMap Live Updates When mounted as a volume, ConfigMap changes are automatically reflected inside the running container within ~60 seconds — without a Pod restart. Environment variable injection does NOT update live — env vars are only read at Pod startup. For config that changes frequently, use volume mounts over env vars.
⚠ Never Commit Raw Secrets to Git base64 is trivially reversible — it is not encryption. Use Sealed Secrets (Bitnami), External Secrets Operator (pulls from AWS Secrets Manager / GCP Secret Manager), or Mozilla SOPS to encrypt secret values before committing. In CI/CD, inject secrets from your secret store at deploy time.
Kubernetes Level 204 — StatefulSets
04
Concept #4

04.StatefulSets — Ordered, Stable Workloads

📖 Definition A StatefulSet manages Pods that require stable, persistent identities. Unlike a Deployment (where Pods are interchangeable with random names), a StatefulSet gives each Pod a stable name (my-db-0, my-db-1), a stable DNS hostname, and a dedicated PersistentVolume that stays bound to that Pod even when it restarts.

StatefulSet vs Deployment — When to Use Each

PropertyDeploymentStatefulSet
Pod identityRandom names (my-app-7d4f-abc)Stable ordered names (my-db-0, my-db-1)
StorageShared or no persistent storageEach Pod gets its own dedicated PVC
Startup orderAll start simultaneouslyOrdered: 0 must be Ready before 1 starts
Shutdown orderAll terminate simultaneouslyReverse order: N, N-1, … 0
Use forStateless apps: APIs, web servers, ML inferenceDatabases, vector stores, message queues
AI exampleRAG API, embedding service, vLLMQdrant, Redis, PostgreSQL, Kafka
💡 A StatefulSet is like a team of specialists at assigned desks — Engineer #0 always sits at Desk 0, has their own filing cabinet (PVC), and colleagues know exactly where to find them. A Deployment is like a pool of temp workers who sit wherever there's a free desk.

StatefulSet YAML — Qdrant Vector Database

qdrant-statefulset.yaml
apiVersion: apps/v1 kind: StatefulSet metadata: name: qdrant namespace: production spec: serviceName: "qdrant" # Headless Service name (required for stable DNS) replicas: 1 selector: matchLabels: app: qdrant template: metadata: labels: app: qdrant spec: containers: - name: qdrant image: qdrant/qdrant:v1.9.0 ports: - containerPort: 6333 # REST API - containerPort: 6334 # gRPC resources: requests: cpu: 500m memory: 4Gi limits: memory: 16Gi volumeMounts: - name: qdrant-storage mountPath: /qdrant/storage volumeClaimTemplates: # KEY DIFFERENCE: each Pod gets its OWN PVC - metadata: name: qdrant-storage spec: accessModes: ["ReadWriteOnce"] resources: requests: storage: 100Gi
Kubernetes Level 205 — DaemonSets
05
Concept #5

05.DaemonSets — One Pod Per Node

📖 Definition A DaemonSet ensures exactly one copy of a Pod runs on every node in the cluster. When a new node joins, the DaemonSet automatically schedules its Pod onto it. When a node is removed, the Pod is garbage-collected. DaemonSets are for infrastructure-level concerns that must be present on every machine.

Classic DaemonSet Use Cases

Log Collectors
Fluentd / Fluent Bit
Every node's container logs need collecting and shipping to Elasticsearch or Loki. One DaemonSet Pod per node tails all container logs on that node.
Node Monitoring
Prometheus node-exporter
CPU, memory, disk, and network metrics need scraping from every node. Node-exporter runs as a DaemonSet, exposing hardware-level metrics for Prometheus.
GPU Device Plugin
NVIDIA device plugin
Lets K8s see and allocate GPUs. Runs as a DaemonSet on GPU nodes. Without it, no Pod can request nvidia.com/gpu resources.

DaemonSet YAML — Fluent Bit Log Collector

fluent-bit-daemonset.yaml
apiVersion: apps/v1 kind: DaemonSet metadata: name: fluent-bit namespace: kube-system # Infrastructure goes in kube-system spec: selector: matchLabels: app: fluent-bit template: metadata: labels: app: fluent-bit spec: tolerations: # Allow scheduling on control plane nodes too - key: node-role.kubernetes.io/control-plane operator: Exists effect: NoSchedule containers: - name: fluent-bit image: fluent/fluent-bit:3.0 volumeMounts: - name: varlog mountPath: /var/log readOnly: true volumes: - name: varlog hostPath: # Mount from the NODE's filesystem (not a PVC) path: /var/log

Taints & Tolerations — Why DaemonSets Need Them

Control plane nodes have a taint (node-role.kubernetes.io/control-plane:NoSchedule) that prevents regular Pods from landing on them. DaemonSets that must run everywhere declare a matching toleration to be scheduled past this taint.

Taint EffectMeaningOverride With
NoScheduleNew Pods will not be placed on this nodeToleration with effect: NoSchedule
PreferNoScheduleK8s tries to avoid the node, uses it if neededOptional toleration
NoExecuteExisting Pods evicted; new Pods won't be placedToleration with effect: NoExecute
Kubernetes Level 206 — Jobs & CronJobs
06
Concept #6

06.Jobs & CronJobs — Batch & Scheduled Tasks

📖 Definitions A Job runs one or more Pods to completion — once all Pods finish successfully, the Job is done. Unlike a Deployment (restarts crashed Pods forever), a Job stops once work is complete. A CronJob schedules a Job on a cron schedule (e.g., nightly at 2am).
Deployment: Run Forever Use for long-running services. If a Pod exits for any reason, the Deployment restarts it immediately. Goal: N replicas always running. Never "done."
Job: Run to Completion Use for finite tasks. Runs until N Pods succeed. If a Pod fails, K8s retries (up to backoffLimit). Once all succeed, Job is done — Pods kept for log inspection, then cleaned up.

Job YAML — Document Indexing Pipeline

indexing-job.yaml
apiVersion: batch/v1 kind: Job metadata: name: index-documents-v2 spec: completions: 1 # How many Pods must succeed parallelism: 1 # How many run simultaneously backoffLimit: 3 # Retry failed Pods up to 3 times activeDeadlineSeconds: 3600 # Kill Job if it runs over 1 hour ttlSecondsAfterFinished: 86400 # Auto-delete 24h after completion template: spec: restartPolicy: OnFailure # Jobs: OnFailure or Never (not Always) containers: - name: indexer image: myregistry/doc-indexer:v3.0 command: ["python", "index.py", "--source=s3://my-docs/"] resources: requests: cpu: 2000m memory: 8Gi

CronJob YAML — Nightly Embedding Refresh

nightly-reindex.yaml
apiVersion: batch/v1 kind: CronJob metadata: name: nightly-reindex spec: schedule: "0 2 * * *" # Run at 2:00 AM every day (UTC) concurrencyPolicy: Forbid # Don't start if previous run still running successfulJobsHistoryLimit: 3 failedJobsHistoryLimit: 1 jobTemplate: spec: template: spec: restartPolicy: OnFailure containers: - name: reindexer image: myregistry/doc-indexer:v3.0 command: ["python", "reindex.py", "--incremental"]
💡 AI Engineering Use Cases for Jobs Jobs are perfect for: one-time document indexing after a major upload, fine-tuning or training runs that terminate on completion, nightly embedding refresh (CronJob), embedding model migration when switching providers, and database schema migrations before a new API deployment.
Kubernetes Level 207 — Init Containers
07
Concept #7

07.Init Containers — Pre-flight Checks

📖 Definition Init containers run and complete successfully before the main container starts. They run in sequence — each must exit with code 0 before the next begins. If any fails, K8s restarts the Pod. Once all complete, they are not run again for that Pod's lifetime.

Problems Init Containers Solve

Wait for DB Loop curl http://qdrant:6333/health until it returns 200. Main container only starts after Qdrant is ready — prevents connection errors on startup.
Download Model Run huggingface-cli download mistralai/Mistral-7B and save to a shared volume. Main vLLM container starts with model already present — no download at runtime.
Run DB Migrations Run alembic upgrade head to migrate the schema. Only after success does the API server start — preventing schema mismatch errors on rollouts.
Set Permissions Run as root to chown a volume directory before the main container (which runs as non-root) mounts it. Main container then has correct write permissions.

Init Container YAML — Model Download + DB Wait

vllm-with-init.yaml · Init containers before main app
spec: initContainers: # All must succeed before main starts - name: wait-for-qdrant # Init 1: wait for vector DB image: curlimages/curl:8.6.0 command: - /bin/sh - -c - | until curl -sf http://qdrant:6333/health; do echo "Waiting..."; sleep 5; done - name: download-model # Init 2: pull model weights image: huggingface/transformers-pytorch-gpu:4.40 command: ["python", "-c"] args: - | from huggingface_hub import snapshot_download snapshot_download("mistralai/Mistral-7B-Instruct-v0.3", local_dir="/model-cache") volumeMounts: - name: model-cache # Shared volume with main container mountPath: /model-cache env: - name: HF_TOKEN valueFrom: secretKeyRef: name: hf-secret key: token containers: # Starts after BOTH init containers succeed - name: vllm image: vllm/vllm-openai:v0.4.2 args: ["--model", "/model-cache"] volumeMounts: - name: model-cache mountPath: /model-cache volumes: - name: model-cache emptyDir: {} # Shared between init and main, Pod-lifetime
Kubernetes Level 208 — Ingress Controllers
08
Concept #8

08.Ingress Controllers — HTTP Routing

📖 Definition An Ingress manages external HTTP/HTTPS access to cluster services. It acts as a smart L7 load balancer — routing traffic based on hostname, URL path, or headers. An Ingress Controller (nginx, Traefik, AWS ALB) is the software that reads Ingress objects and implements the routing.

LoadBalancer Service vs Ingress

AspectLoadBalancer ServiceIngress
LayerL4 (TCP/UDP) — no HTTP awarenessL7 (HTTP) — sees URLs, headers, hostnames
Path-based routing❌ Not possible✅ /api → svc A, /docs → svc B
Host-based routing❌ Not possible✅ api.app.com vs admin.app.com
TLS terminationPer-service, manual✅ Single point with cert-manager
Cloud costOne cloud LB per Service (expensive)One cloud LB for ALL services (economical)
Rate limiting❌ Not possible✅ Via nginx annotations
💡 A LoadBalancer Service is like a separate entrance door for every office in a building. An Ingress is a single main entrance with a receptionist who reads your badge and directs you to the right office.

Ingress YAML — Multi-Service RAG Application

rag-ingress.yaml · Path-based routing + TLS
apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: rag-ingress namespace: production annotations: kubernetes.io/ingress.class: "nginx" nginx.ingress.kubernetes.io/ssl-redirect: "true" nginx.ingress.kubernetes.io/limit-rps: "100" cert-manager.io/cluster-issuer: "letsencrypt-prod" spec: tls: - hosts: - api.myragapp.com secretName: rag-tls-cert # cert-manager stores TLS cert here rules: - host: api.myragapp.com http: paths: - path: /v1/query pathType: Prefix backend: service: name: rag-api-svc port: number: 80 - path: /v1/ingest pathType: Prefix backend: service: name: ingest-svc port: number: 80 - path: / pathType: Prefix backend: service: name: frontend-svc port: number: 80
💡 cert-manager — Automatic Free TLS Install cert-manager and it automatically provisions and renews Let's Encrypt TLS certificates for any Ingress with the cert-manager.io/cluster-issuer annotation. No manual certificate management needed.
Kubernetes Level 209 — Network Policies
09
Concept #9

09.Network Policies — Pod-Level Firewalls

📖 Definition A NetworkPolicy controls which Pods can send traffic to which other Pods. By default, Kubernetes has no network restrictions — every Pod can reach every other Pod. NetworkPolicies add firewall rules at the Pod level, enabling zero-trust networking.
⚠ Why This Matters for Security Without NetworkPolicies: if an attacker compromises your frontend Pod, they can directly reach your vector database, model API, and admin services. Nothing blocks lateral movement inside the cluster. NetworkPolicies implement least-privilege networking.

NetworkPolicy YAML — Deny All Then Allow Selectively

network-policies.yaml · Zero-trust pattern
apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: default-deny-all # Step 1: lock down the namespace namespace: production spec: podSelector: {} # Empty = select ALL pods in namespace policyTypes: - Ingress - Egress # No ingress/egress rules = deny everything --- apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: allow-rag-to-qdrant # Step 2: selectively allow namespace: production spec: podSelector: matchLabels: app: qdrant # Policy applies TO qdrant Pods policyTypes: - Ingress ingress: - from: - podSelector: matchLabels: app: rag-api # Only rag-api Pods can reach qdrant ports: - protocol: TCP port: 6333
⚠ NetworkPolicy Requires a Supported CNI NetworkPolicies only work with a CNI that enforces them: Calico, Cilium, or Weave Net. The default CNI in many clusters (Flannel, kubenet) silently ignores NetworkPolicies — they'll exist but do nothing. Check your CNI before relying on network isolation.
❌ Common Pitfall: Forgetting DNS Egress After applying default-deny-all, your Pods can no longer resolve DNS names. They can't reach qdrant-svc, can't call OpenAI, can't reach anything. Always add an egress rule allowing port 53 (UDP/TCP) to kube-dns in kube-system after locking down a namespace.
Kubernetes Level 210 — RBAC
10
Concept #10

10.RBAC — Role-Based Access Control

📖 Definition RBAC controls which users, groups, or Pods (via ServiceAccounts) can perform which actions (get, list, create, delete) on which Kubernetes resources (pods, secrets, deployments). It enforces least-privilege access to your cluster.

The Four RBAC Objects

Role
Namespace-scoped permissions
Grants permissions within one namespace. "In namespace production, allow get/list pods."
RoleBinding
Attaches a Role to a subject
Binds a Role to a User, Group, or ServiceAccount. "Give 'pod-reader' Role to ServiceAccount 'deployer'."
ClusterRole
Cluster-wide permissions
Like Role but applies across all namespaces or to cluster-scoped resources like Nodes.
ServiceAccount
Identity for Pods
An identity for Pods to use when calling the K8s API. Every Pod runs as a ServiceAccount (default or custom).

RBAC YAML — CI/CD Pipeline Access

rbac.yaml · ServiceAccount + Role + Binding
apiVersion: v1 kind: ServiceAccount metadata: name: deployer namespace: production --- apiVersion: rbac.authorization.k8s.io/v1 kind: Role metadata: name: deployment-manager namespace: production rules: - apiGroups: ["apps"] resources: ["deployments"] verbs: ["get", "list", "update", "patch"] - apiGroups: [""] # "" = core API group (Pods, Services) resources: ["pods"] verbs: ["get", "list"] --- apiVersion: rbac.authorization.k8s.io/v1 kind: RoleBinding metadata: name: deployer-binding namespace: production subjects: - kind: ServiceAccount name: deployer namespace: production roleRef: kind: Role name: deployment-manager apiGroup: rbac.authorization.k8s.io
kubectl · RBAC Debugging
$ kubectl auth can-i get pods --as=system:serviceaccount:production:deployer $ kubectl get rolebindings -n production $ kubectl describe rolebinding deployer-binding -n production
Kubernetes Level 211 — Security Contexts
11
Concept #11

11.Security Contexts & Pod Security

📖 Definition A securityContext defines privilege and access control settings for a Pod or container. It lets you run containers as non-root users, make filesystems read-only, drop Linux capabilities, and prevent privilege escalation — hardening your containers against exploits.

Key Security Context Fields

FieldLevelWhat It DoesBest Practice
runAsNonRoot: truePod/ContainerK8s refuses to start if image runs as root (UID 0)Always set for production
runAsUser: 1000Pod/ContainerOverride the UID the container process runs asUse 1000+ (non-root range)
readOnlyRootFilesystem: trueContainerRoot filesystem mounted read-only — blocks malware writesUse + volumeMount for writable dirs
allowPrivilegeEscalation: falseContainerProcess cannot gain more privileges than its parentAlways false for app containers
capabilities.drop: ["ALL"]ContainerDrop all Linux capabilities; add back only what's neededDrop ALL, then add individually
seccompProfile: RuntimeDefaultPodEnables default seccomp filter — blocks dangerous syscallsUse RuntimeDefault as baseline

Hardened Container Security Context

Pod spec · Hardened security context
spec: securityContext: # Pod-level: applies to all containers runAsNonRoot: true runAsUser: 1000 runAsGroup: 1000 fsGroup: 1000 # All volume files owned by GID 1000 seccompProfile: type: RuntimeDefault containers: - name: rag-api image: myregistry/rag-api:v2 securityContext: # Container-level overrides readOnlyRootFilesystem: true allowPrivilegeEscalation: false capabilities: drop: ["ALL"] volumeMounts: - name: tmp-dir # readOnly root FS needs explicit writable /tmp mountPath: /tmp - name: cache-dir mountPath: /app/.cache volumes: - name: tmp-dir emptyDir: {} - name: cache-dir emptyDir: {}
💡 Pod Security Admission (K8s 1.25+) Label your namespace to enforce security standards cluster-wide: pod-security.kubernetes.io/enforce: restricted automatically rejects any Pod that runs as root or lacks seccomp — no per-Pod annotation needed.
Kubernetes Level 212 — Helm
12
Concept #12

12.Helm — The Kubernetes Package Manager

📖 Definition Helm is the package manager for Kubernetes. A Chart is a package of templated YAML files configurable via values.yaml. When deployed, a chart becomes a Release. Helm makes it trivial to install community software (Qdrant, Redis, PostgreSQL, nginx-ingress) and to package your own apps for repeatable deployment.

Core Helm Concepts

Chart
The package / blueprint
A directory of YAML templates + values.yaml with defaults + Chart.yaml with metadata. Published to Helm repos (ArtifactHub, private registries).
Values
The configuration knobs
YAML parameters (image tag, replica count, resource limits) that override chart defaults. Maintain separate values files per environment: dev, staging, prod.
Release
A deployed instance of a chart
When you helm install, Helm creates a named, versioned Release with rollback history. Multiple releases of the same chart can exist in different namespaces.

Essential Helm Commands

Terminal · Helm Commands
── REPOS ──────────────────────────────────────────────────────────────── $ helm repo add qdrant https://qdrant.github.io/qdrant-helm $ helm repo update # Fetch latest chart versions $ helm search repo qdrant ── INSTALL / UPGRADE ──────────────────────────────────────────────────── $ helm install my-qdrant qdrant/qdrant -n production -f prod-values.yaml $ helm upgrade my-qdrant qdrant/qdrant -n production -f prod-values.yaml $ helm upgrade --install my-qdrant qdrant/qdrant # Install OR upgrade in one cmd ── INSPECT ────────────────────────────────────────────────────────────── $ helm list -n production $ helm history my-qdrant -n production # All revisions with timestamps $ helm get values my-qdrant -n production # What values are applied now $ helm template my-qdrant qdrant/qdrant -f prod-values.yaml # Render YAML locally without touching the cluster — great for debugging ── ROLLBACK / CLEANUP ─────────────────────────────────────────────────── $ helm rollback my-qdrant 2 # Roll back to revision 2 $ helm uninstall my-qdrant -n production # Remove release + all K8s resources

prod-values.yaml — Override Chart Defaults

prod-values.yaml
replicaCount: 1 image: tag: "v1.9.0" # Pin a specific version resources: requests: memory: "4Gi" cpu: "500m" limits: memory: "16Gi" persistence: enabled: true size: "100Gi" storageClass: "gp3" # AWS gp3 SSD
Kubernetes Level 213 — Writing Your Own Helm Chart

Concept #13

13.Writing Your Own Helm Chart

Chart Directory Structure

rag-app/ · Helm chart structure
rag-app/ ├── Chart.yaml # Chart metadata: name, version, description ├── values.yaml # Default values — users override these ├── values-staging.yaml # Staging-specific overrides ├── values-prod.yaml # Production-specific overrides ├── templates/ │ ├── deployment.yaml # Templated Deployment (uses {{ .Values.xxx }}) │ ├── service.yaml │ ├── ingress.yaml │ ├── configmap.yaml │ ├── hpa.yaml │ └── _helpers.tpl # Named template helpers (reusable snippets) └── charts/ # Sub-charts / dependencies (qdrant, redis)

Chart.yaml

Chart.yaml
apiVersion: v2 name: rag-app description: RAG application — API, embeddings, and vector store type: application version: 1.3.0 # Chart version — increment on chart changes appVersion: "2.1.0" # Your app version — informational dependencies: - name: qdrant # Include Qdrant as a sub-chart version: "0.8.4" repository: "https://qdrant.github.io/qdrant-helm"

Templated Deployment — Go Template Syntax

templates/deployment.yaml
apiVersion: apps/v1 kind: Deployment metadata: name: {{ include "rag-app.fullname" . }} # Helper from _helpers.tpl namespace: {{ .Release.Namespace }} # Auto-filled by Helm spec: replicas: {{ .Values.replicaCount }} # From values.yaml template: spec: containers: - name: rag-api image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}" resources: requests: cpu: {{ .Values.resources.requests.cpu }} memory: {{ .Values.resources.requests.memory }} {{- if .Values.hpa.enabled }} # Conditional blocks # HPA resource only rendered if hpa.enabled=true {{- end }}
Terminal · Full stack deploy with Helm
$ helm dependency update ./rag-app # Pull sub-charts (qdrant, redis) $ helm upgrade --install rag-app ./rag-app \ --namespace production --create-namespace \ -f values-prod.yaml \ --set image.tag=v2.3.1 \ # Override from CI/CD pipeline --wait \ # Block until all Deployments ready --timeout 10m # Give model-loading Pods enough time
💡 helm template for Local Debugging Run helm template my-release ./rag-app -f values-prod.yaml to render all templates locally without touching the cluster. Pipe to kubectl apply --dry-run=client -f - for a full validation check.
Kubernetes Level 214 — Advanced Probes & Graceful Shutdown
14
Concept #14

14.Advanced Probes & Graceful Shutdown

Graceful Shutdown for LLM Inference Servers

When K8s needs to stop a Pod (scale-down, rolling update, node drain), it sends SIGTERM. The container has a grace period to finish in-flight requests before being forcefully killed (SIGKILL). Default: 30 seconds. For LLM inference, a single generation request can take 45–120 seconds — you must increase this.

Pod spec · Graceful shutdown for LLM servers
spec: terminationGracePeriodSeconds: 120 # 2 min for in-flight completions containers: - name: vllm lifecycle: preStop: # Runs BEFORE SIGTERM is sent exec: command: ["/bin/sh", "-c", "sleep 5"] # 5s delay lets the LB stop sending new requests here postStart: # Runs immediately after container starts exec: command: ["python", "warmup.py"]

Startup Probe — For Slow-Starting Models

Container spec · Startup probe for LLM server
startupProbe: # Disables liveness/readiness until this passes httpGet: path: /v1/models port: 8000 failureThreshold: 40 # 40 × 15s = 10 minutes max startup time periodSeconds: 15 livenessProbe: httpGet: path: /v1/models port: 8000 periodSeconds: 30 failureThreshold: 3 # Restart after 3 consecutive failures timeoutSeconds: 5 readinessProbe: httpGet: path: /v1/models port: 8000 periodSeconds: 10 failureThreshold: 1 # Remove from LB after just 1 failure

Pod Termination Sequence

1
Pod marked Terminating
Pod removed from Service endpoints — no new requests routed to it
2
preStop hook runs
Your preStop command executes (e.g., sleep 5 to let LB propagate)
3
SIGTERM sent to container
Container should start graceful shutdown and finish in-flight requests
4
Grace period counts down
terminationGracePeriodSeconds timer runs (set to 120 for LLM servers)
5
SIGKILL if still running
If container hasn't exited by end of grace period, K8s force-kills it

15.Resource Quotas & LimitRanges

📖 Definitions A ResourceQuota sets a hard limit on total resources an entire namespace can consume. A LimitRange sets default and maximum resource requests/limits for individual containers. Together they prevent any team or namespace from monopolising cluster resources.

ResourceQuota YAML

production-quota.yaml · Namespace budget
apiVersion: v1 kind: ResourceQuota metadata: name: production-quota namespace: production spec: hard: requests.cpu: "40" # Max total CPU requests: 40 cores requests.memory: 80Gi limits.cpu: "80" limits.memory: 160Gi pods: "50" # Max 50 Pods in namespace services.loadbalancers: "2" # Max 2 cloud LBs requests.nvidia.com/gpu: "4" # Max 4 GPUs

LimitRange YAML

limitrange.yaml · Per-container defaults & caps
apiVersion: v1 kind: LimitRange metadata: name: default-limits namespace: production spec: limits: - type: Container defaultRequest: # Applied if container omits requests cpu: 100m memory: 128Mi default: # Applied if container omits limits cpu: 500m memory: 512Mi max: # Hard cap — no container may exceed these cpu: 8000m memory: 32Gi
AspectResourceQuotaLimitRange
ScopeTotal for the whole namespacePer individual container or Pod
PurposePrevent namespace from exceeding a budgetProvide safe defaults; enforce per-container caps
Effect when exceededNew object creation rejected (quota exceeded)Container rejected if it exceeds max limits
⚠ ResourceQuota + LimitRange Interaction If a namespace has a ResourceQuota, all containers must specify resource requests and limits. If a container omits them and there is no LimitRange to supply defaults, K8s refuses to create the Pod. Always deploy a LimitRange alongside a ResourceQuota.

Common Level 2 Mistakes

❌ Using :latest Tag in Production You push a broken build, it overwrites latest, K8s pulls it on the next restart. No specific version to roll back to. Fix: Always pin: image: myregistry/rag-api:v2.3.1
⚠ Deployment Instead of StatefulSet for Vector DB K8s reschedules the Pod to a different node. RWO PVC might not follow. Qdrant comes up empty. Fix: Use StatefulSet with volumeClaimTemplates for any stateful service.
⚠ 30s Grace Period for LLM Inference vLLM is killed mid-generation on rolling updates. Users get cut-off responses. Fix: Set terminationGracePeriodSeconds to 90–180s + preStop sleep 5.
⚠ Init Container Not Idempotent Migration init container runs, Pod crashes, K8s restarts Pod, migration runs again and fails. Fix: Use Alembic/Flyway that track which migrations have been applied.
